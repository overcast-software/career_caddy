** Dispatch protocol
1. =cc-orchestrator= reads =todo.org=, picks the next claimable heading (state =TODO= under Backlog or Active; Inbox items get triaged first via =cc-todo-move=).
2. Edit the heading on =main=: flip =TODO= → =STRT=, set =:CLAIMED:= to today's date and =:OWNER:= to the agent id (=cc-api=, =cc-frontend=, =cc-auto=, or self for parent-scope work).
3. Commit on =main= (parent or submodule, whichever repo owns the heading): =STRT: <task title>=. That commit is the dispatch lock.
4. Invoke the =Agent= tool with =subagent_type: "<owner>"=, =isolation: "worktree"=, and a self-contained brief (peer-file paths, acceptance, lint/test commands).
5. Sub-agent works in its worktree, runs file-level lint + focused tests, commits, returns worktree path + branch.
6. =cc-orchestrator= reviews =git -C <repo> diff main..agent/task-NNN=, opens PR via =gh pr create -R overcast-software/career_caddy_<sub>=, merges with =--merge --delete-branch=false= (so the SHA the parent pin will reference stays reachable from main).
7. **Submodule merge order is non-negotiable**: child first → parent pin bump last, always. Parent merge is its own PR after the child PR has landed.
8. Cleanup: =git -C <repo> checkout main && pull --ff-only && branch -d agent/task-NNN && push origin --delete agent/task-NNN && worktree remove <root>/task-NNN=.

| =KILL=    | Dropped from scope                                           | Anyone                                              |

** Race-avoidance
The =STRT= commit on =main= is the lock. Before dispatching:
- =git -C <repo> log --oneline -5 main= — look for a recent =STRT: <task>= line that matches.
- =git -C <repo> worktree list= — confirm no active worktree on that branch.
If a heading is already =STRT= and a worktree exists, do NOT dispatch a fresh agent — either resume the existing worktree or wait. Never two worktrees on the same heading.

** When work crosses repo boundaries
Some changes touch multiple repos (e.g. API contract change ripples into frontend and cc_auto). =cc-orchestrator= splits these into separate per-repo headings under one umbrella, dispatches them in dependency order, and only bumps the parent pin after all child PRs have merged. Each per-repo heading carries its own =:OWNER:= so the dispatch protocol's per-agent claim still applies.

Consumer: =components/scrapes/item.js _loadScreenshotList()= called from the constructor (already a separate code-smell — violates =feedback_no_logic_in_component_constructors=).

***** Pros of refactor
- Consumer becomes a pure renderer over =scrape.screenshotsList= — kills the constructor side-effect at the same time.
- Scrape model already declares =scrapeStatuses= hasMany; =screenshots= as a sibling matches the established pattern on the same parent.
- Single round-trip: =scrapes/show= already does =include: 'company,job-post,scrape-statuses'=. Adding =,screenshots= piggybacks on the existing fetch with zero new requests.

***** Cons of refactor
- Screenshots are *staff-only* and frequently empty for non-debug scrapes. Eager sideload via =?include=screenshots= adds an extra DB round-trip per scrape view even when the user never expands the dropdown. Today's lazy-load is on-demand and skipped entirely for non-staff.

- The =scrapes/show= include= line gets longer and tied to a staff-only relationship; non-staff requests would need to omit =screenshots= or the API needs to no-op the include for non-staff users (extra branching).

***** Recommendation
