** TODO Deep-link navigate targets into resume edit form (anchors)  :APPROVED:
[2026-04-18 ai+frontend] Follow-on from the "no editing linked
resources via chat" decision. When chat sends a user to
=/resumes/{id}/edit= to fix something, it currently drops them at
the top of a long form. Better: deep-link to the specific field.

Three options (cheapest → richest):
1. *Hash-fragment scroll* — render =id="experience-{id}"= (and similar
   anchors for education / certification / project) on each card in
   =resumes/edit.hbs=. Agent emits
   ={navigate: "/resumes/39/edit#experience-128"}=; browser scrolls.
   Minimal change; no Ember focus logic.
2. *Query-param focus* — =?focus=experience-128.title= on the edit
   route. Route handler expands the matching panel + calls =.focus()=
   on the target input after render. Needs a small helper mapping
   =resource.field= → DOM input id.
3. *Field-level anchors* — =#experience-128-title=, =#education-77-
   degree=, etc. Hybrid of (1) and (2). Agent already knows the
   resource id and field name from =show_resume= markdown.

Prompt surface: teach the agent "when user wants to change an
experience field, emit
={navigate: '/resumes/{r}/edit?focus=experience-{e}.{field}'}=";
edit route handles the rest. No new tools.

Disambiguation risk: if the user says "change my title at RHI" and
has two Robert Half experiences, the agent still has to pick. Same
problem as today — this TODO is about delivery, not selection.

Value:
- User edits themselves → fewer silent-overwrite bugs like the
  OFFICE BADASS incident.
- Deep-link IS the teaching moment; next time they know where the
  form lives.
- Zero new chat tools to maintain.

** TODO Main chat: "intent announced, tool skipped" (systemic) :APPROVED:
[2026-04-18 ai] Recurring pattern on haiku-4-5 in the main chat: model
emits "I'll check X" / "I'll create Y" / "Let me look that up" — then
the turn ends with no tool call. Step B already fixed this for
onboarding by forcing delegation. Remaining examples seen in e2e:
- User pasted LinkedIn job markup, AW said "I'll create the job post"
  → no POST to =/api/v1/job-posts/=.
- User asked "what should I do next?" with all onboarding steps done
  (no AW block injected) → "I'll check what job posts you have"
  → no =get_job_posts= call.

Both hit the same systemic gap: the main chat prompt tells the agent
*what* to do but doesn't force *how*. Negative-rules-in-a-big-prompt
continues to be the weakness.

Candidate fixes (pick one, not all):
1. *Tightened tool-use directive at the top of SYSTEM_PROMPT*, similar
   to the AW delegation rule: "When you say 'I'll check X' or 'I'll
   create Y', you MUST have called the corresponding tool in THIS
   turn. If you don't call it, don't announce it."
2. *Job-post creation sub-agent* (mirrors step B): a dedicated
   pydantic-ai agent owns the pasted-text → structured extraction →
   create flow, with a tight prompt and 3 tools. Main chat delegates
   via a =create_job_post_from_text= delegation tool. Worth it if we
   expect users to paste job descriptions routinely.
3. *Just-in-time prompt rider*: when the user message exceeds N chars
   OR contains job-post-like keywords (title / salary / location),
   _build_system_prompt adds a focused "MUST extract and call
   create_job_post_with_company_check" stanza.

Order of attack: try #1 first (cheap, 5-line prompt tweak). If
pasted-text creates still skip the tool call, escalate to #2.

** TODO Canonicalize tracker URLs on scrape ingest [api]
[2026-04-18] Same root cause as the automation-side dedup bug: marketing
emails (and, crucially, user-pasted links from those emails) use
SendGrid-style click trackers — e.g. =url9751.alerts.jobot.com/ls/click?upn=…=,
=click.ziprecruiter.com=, LinkedIn =/comm/…= with =trackingId/refId=,
Mailgun =email.mg.…=. Each link is unique per recipient but redirects to
the same destination, so =filter[link]= dedup on the raw string misses
them. Three real examples observed: two distinct tracker URLs → same
job, one tracker URL → 404 (dead; we'd still create a post).

When the API receives a scrape / job-post create request:
1. If the link's host matches a known tracker pattern, resolve it with
   a HEAD + follow-redirects (3s timeout). Store the resolved URL as
   canonical.
2. Strip tracking query params from the resolved URL (=utm_*=,
   =trackingId=, =refId=, =gclid=, =fbclid=, =midToken=, =otpToken=,
   =lipi=, =trk*=, =eid=, =otpToken=).
3. Dedup on the canonical URL, not the submitted URL. Keep the original
   in a =source_link= column so we can debug where the post came from.
4. If the tracker returns 4xx/5xx, reject the scrape with a clear error
   rather than silently creating a dead post.

Coordinate with the automation-side fix (=src/agents/url_extractor.py=)
so both produce the same canonical form — otherwise the dedup still
breaks when an email link and a user-pasted link reach the API through
different paths.
