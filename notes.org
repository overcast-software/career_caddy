*** Decision [2026-04-08 product-owner]
- Multi-tenancy: APPROVED.
- Ownership model: use =is_staff= as the superuser flag.
- No new =owner= FK needed — leverage existing fields (confirmed below).

*** API notes [api]
No new FK column needed on any model. Every endpoint already has a user reference; the field name just varies.
Enforce at the queryset level per ViewSet using a class attribute =user_lookup= that names the ORM filter expression:

| ViewSet | Existing field | queryset filter |
|---|---|---|
| =CoverLetterViewSet= | =user= | =user_id=request.user.id= |
| =JobApplicationViewSet= | =user= | =user_id=request.user.id= |
| =ResumeViewSet= | =user= | =user_id=request.user.id= |
| =ScoreViewSet= | =user= | =user_id=request.user.id= |
| =SummaryViewSet= | =user= | =user_id=request.user.id= |
| =ProjectViewSet= | =user= | =user_id=request.user.id= |
| =JobPostViewSet= | =created_by= | =created_by_id=request.user.id= |
| =QuestionViewSet= | =created_by= | =created_by_id=request.user.id= |
| =AnswerViewSet= | none direct | =question__created_by_id=request.user.id= |
| =ScrapeViewSet= | none direct | =job_post__created_by_id=request.user.id= |
| =CompanyViewSet= | none | intentionally shared — no filter |
| Skill, Status | none | global lookup tables — no filter |
| Pivot tables (ResumeSKill etc.) | none | scoped via parent endpoint only |

Mixin approach (one place, covers most ViewSets):
#+BEGIN_SRC python
class OwnedQuerysetMixin:
    user_lookup = "user_id"          # override per ViewSet where needed
    def get_queryset(self):
        qs = super().get_queryset()
        if self.request.user.is_staff:
            return qs
        return qs.filter(**{self.user_lookup: self.request.user.id})
#+END_SRC

ViewSets with a different field name just set =user_lookup = "created_by_id"= or ="question__created_by_id"=.
No model migrations. No new columns.

** Lazy load child resources
[[file:todo.org::*Lazy load the child resources][↑ todo.org]]

- Currently routes eagerly include related records (e.g. =include: 'job-post,company'= on scrapes show).
- Removing includes means the page renders first, then fires secondary =findRecord= calls on demand.
- User-visible impact: some detail sections will be blank briefly until the secondary fetch resolves.
- Multi-tenancy benefit: pre-loading cross-owner data becomes impossible, reducing data-leak risk.
- Approach: audit each =include= in Ember routes, classify as load-bearing (required for render) vs. incidental (only used in one conditional block).

** Dockerfile caching
[[file:todo.org::*Caching deploy steps so it goes faster][↑ todo.org]]

- Standard optimisation: copy dependency manifests (=pyproject.toml=, =uv.lock= for API; =package.json= for frontend) before source code so the dependency layer is cached between rebuilds.
- Self-contained, zero risk, no design decisions required.
- Expected improvement: rebuilds after source-only changes drop from ~3 min to ~30 sec.

*** API notes [api]
- Current API Dockerfile copies the entire source tree before =uv sync=, busting the dependency cache on every code change.
- Fix: copy =pyproject.toml= + =uv.lock= → run =uv sync --frozen --no-install-project= → copy source → run =uv sync= again. Two-stage install, cache-friendly.
- The =--no-install-project= flag installs dependencies only (not the app itself), so the heavy layer is cached as long as lockfile doesn't change.
- Dagger pipeline in =dagger/src/main.py= may also benefit from the same reordering.
- No risk to production behaviour; purely build-time change.

** Remove SQLAlchemy from API
[[file:todo.org::*Completely remove SQLAlchemy from api][↑ todo.org]]

- The API uses SQLAlchemy for all 33 domain models; Django ORM only handles auth tables.
- Removing it means rewriting all 33 model classes, replacing SQLAlchemy sessions with Django ORM, and re-running migrations.
- Estimated effort: large (multi-week API branch). No user-facing impact. Reduces stack complexity significantly.
- Low urgency; the current dual-ORM setup works, it's just technical debt.

*** API notes [api]
- A migration plan already exists at =api/job_hunting/SA_REMOVAL_PLAN.md= — read it before starting.
- Current state: all 33 domain models are already Django ORM models (the SA removal is largely done at the model layer). The remaining SA usage is in =lib/db.py= (session management), =lib/models/base.py= (BaseModel mixin), and a handful of =get_session()= calls in ViewSets.
- Concrete remaining work: delete =lib/db.py=, remove =BaseModel.get_session()= calls, remove =get_session= overrides from ViewSets, drop =sqlalchemy= from =pyproject.toml=.
- Risk: low — if the model layer is already Django ORM, the session removal is mechanical.
