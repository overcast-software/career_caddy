** Reactive Resume as a PDF-render plugin
[[file:todo.org::*Reactive Resume plugin][↑ todo.org]]

Use a self-hosted [[https://rxresu.me][Reactive Resume]] on =resume.careercaddy.online= strictly
as a *PDF rendering backend* — Career Caddy owns the resume data, rxresume
owns the themes and the =pdf= pipeline. Ship the integration as a *separate
pip-installable plugin* so the core app's scope does not widen.

*** Scope boundaries (locked)
- No schema changes in core. No =rxresume_id= column, no new tables, no
  migration.
- No bidirectional sync. No per-user accounts on rxresume. No OAuth.
- The rxresume resource is *transient*: import → print → delete, all in one
  request. Nothing persisted on the rxresume side after the request finishes.
- One service-account user and one API key on the rxresume side, both owned
  by the plugin's env (=RXRESUME_API_KEY=), not by core.

*** Why a plugin and not an in-app feature
- Keeps core narrow — the whole rxresume concept is absent unless the plugin
  is =pip install='d.
- Lets us swap renderers later (WeasyPrint, Typst, rxresume, something else)
  without touching core.
- Matches the /career_caddy_automation/ precedent of extracting optional
  subsystems out of the main repo when they start carrying their own weight.

*** Plugin framework — [[https://djp.readthedocs.io][djp]]
- Simon Willison's [[https://github.com/pytest-dev/pluggy][pluggy]] wrapper built for Django. Plugins are pip
  packages discovered via setuptools entry points.
- Hooks djp already provides that we'd use: =installed_apps=, =urlpatterns=,
  =settings=.
- Core adds one custom hookspec for renderers:
  #+begin_src python
  # core: api/plugins/hookspecs.py
  import djp
  hookspec = djp.hookspec
  @hookspec
  def resume_renderers():
      """Return dict[str, ResumeRenderer] keyed by render format (e.g. 'pdf')."""
  #+end_src
- Fallback if djp is too much: plain entry points
  (=[project.entry-points."career_caddy.renderers"]=) and ~30 lines of
  discovery in core. Same UX, no dep.

*** Transient-resource flow (no DB)
1. User clicks "Download PDF" on =resumes/show=.
2. Core view loads the =Resume=, checks ownership, hands it to the renderer.
3. Plugin =POST='s =/resume/import= → short-lived =rxresume_id= in a local var.
4. Plugin =GET='s =/api/resume/print/{id}= → PDF bytes.
5. Plugin =DELETE='s the transient rxresume resource (no service-account garbage).
6. Core streams the bytes to the browser.

*** Plugin package layout (separate repo)
=cc-rxresume-plugin/=
- =pyproject.toml= declares the djp entry point:
  #+begin_src toml
  [project.entry-points.djp]
  cc_rxresume = "cc_rxresume.plugin"
  #+end_src
- =cc_rxresume/plugin.py= — djp hookimpls: register the renderer; contribute
  no URLs (core's render endpoint is enough).
- =cc_rxresume/renderer.py= — =ReactiveResumeRenderer= (stateless service
  object). Signature sketch:
  #+begin_src python
  class ReactiveResumeRenderer:
      def __init__(self, base_url: str, api_key: str, theme: str = "azurill"): ...
      def render_pdf(self, resume: "Resume") -> bytes:
          data = self._to_rxresume_json(resume)
          rxid = self._import(data)
          try:
              return self._print(rxid)
          finally:
              self._delete(rxid)
  #+end_src
- =cc_rxresume/mapper.py= — pure functions that translate a Career Caddy
  =Resume= (via =to_export_context()=, resume.py:135) into rxresume's
  =ResumeData= schema. Validates against =GET /api/resume/schema= in tests.
- =cc_rxresume/client.py= — httpx wrapper with =x-api-key= header, following
  the pattern in [[file:ai/lib/api_tools.py][ai/lib/api_tools.py]].
- Env it reads: =RXRESUME_BASE_URL=, =RXRESUME_API_KEY=, =RXRESUME_THEME=.

*** Core changes (minimal)
- Add =djp= to =api/pyproject.toml= and wire in =api/config/settings.py=:
  =INSTALLED_APPS += djp.installed_apps()=, =urlpatterns += djp.urlpatterns()=.
- Add =api/plugins/hookspecs.py= with the =resume_renderers= hookspec.
- Add one generic endpoint =GET /api/v1/resumes/:id/render/:format/= in
  [[file:api/job_hunting/api/views/career_data.py][career_data.py]] that:
  - Checks ownership (=resume.user == request.user=).
  - Looks up the renderer for =:format= via the plugin manager.
  - Returns =404= if no plugin supplies that format — core has no hardcoded
    knowledge of rxresume.
  - Streams the bytes.
- No model, serializer, or migration changes. Net diff in core: ~60 lines.

*** Deployment presumptions

** Multi-tenancy — is_staff role + per-queryset ownership filter :IMPORTANT:
[[file:todo.org::*As a superuser I want to be able to manage user accounts][↑ todo.org]]

- Currently no row-level ownership on the 33 domain models — any authenticated user can read any record.
- Multi-tenancy requires adding an =owner= FK on every model and filtering every queryset.
- Implementation options:
  - Django custom =get_queryset= on every ViewSet (surgical, low magic).
  - A shared mixin =OwnedQuerysetMixin= that enforces =owner=request.user= automatically.
- Superuser role: Django =is_staff= flag or a dedicated =role= field on the User model?
  is staff is fine
- This is a large API-first feature; frontend follows once the ownership model is defined.

*** Decision
:PROPERTIES:
:DECIDED:  [2026-04-08 product-owner]
:END:
- Multi-tenancy: APPROVED.
- Ownership model: use =is_staff= as the superuser flag.
- No new =owner= FK needed — leverage existing fields (confirmed below).

*** API notes [api]
No new FK column needed on any model. Every endpoint already has a user reference; the field name just varies.
Enforce at the queryset level per ViewSet using a class attribute =user_lookup= that names the ORM filter expression:

| ViewSet | Existing field | queryset filter |
|---|---|---|
| =CoverLetterViewSet= | =user= | =user_id=request.user.id= |
| =JobApplicationViewSet= | =user= | =user_id=request.user.id= |
| =ResumeViewSet= | =user= | =user_id=request.user.id= |
| =ScoreViewSet= | =user= | =user_id=request.user.id= |
| =SummaryViewSet= | =user= | =user_id=request.user.id= |
| =ProjectViewSet= | =user= | =user_id=request.user.id= |
| =JobPostViewSet= | =created_by= | =created_by_id=request.user.id= |
| =QuestionViewSet= | =created_by= | =created_by_id=request.user.id= |
| =AnswerViewSet= | none direct | =question__created_by_id=request.user.id= |
| =ScrapeViewSet= | none direct | =job_post__created_by_id=request.user.id= |
| =CompanyViewSet= | none | intentionally shared — no filter |
| Skill, Status | none | global lookup tables — no filter |
| Pivot tables (ResumeSKill etc.) | none | scoped via parent endpoint only |

Mixin approach (one place, covers most ViewSets):
#+BEGIN_SRC python
class OwnedQuerysetMixin:
    user_lookup = "user_id"          # override per ViewSet where needed
    def get_queryset(self):
        qs = super().get_queryset()
        if self.request.user.is_staff:
            return qs
        return qs.filter(**{self.user_lookup: self.request.user.id})
#+END_SRC

ViewSets with a different field name just set =user_lookup = "created_by_id"= or ="question__created_by_id"=.
No model migrations. No new columns.

** Pydantic-AI toolsets — dynamic tool scoping
:PROPERTIES:
:CREATED:  [2026-04-10]
:END:
[[file:todo.org::*MCP exposure][↑ todo.org]]

*** Problem
All pydantic-ai agents connect to =career_caddy_server.py= via =MCPServerStdio=, spawning a full Python subprocess per agent run. Every agent gets the same 14 tools regardless of what it actually needs. This adds latency, memory overhead, and prevents fine-grained tool scoping.

*** Solution: =CareerCaddyToolset(AbstractToolset)=
A single custom toolset class with a =scope= parameter that selects which =lib/api_tools.py= functions to expose. No subprocess, no MCP protocol overhead — direct in-process calls.

#+BEGIN_SRC python
from lib.toolsets import CareerCaddyToolset, CareerCaddyDeps

# Scoped to just what this agent needs
agent = Agent(
    "openai:gpt-4o-mini",
    deps_type=CareerCaddyDeps,
    toolsets=[CareerCaddyToolset(scope="scoring")],
)
result = await agent.run(prompt, deps=CareerCaddyDeps(api_token=token))
#+END_SRC

*** Deps type
#+BEGIN_SRC python
@dataclass
class CareerCaddyDeps:
    api_token: str
    base_url: str = "http://localhost:8000"
#+END_SRC

The toolset builds an =ApiClient(deps.base_url, deps.api_token)= from =ctx.deps= at call time and passes it to each =api_tools.*= function.

*** Scopes
| Scope | Tools | Use case |
|---|---|---|
| =all= | All 19 functions | Full access |
| =career_caddy= | Original 14 | Drop-in replacement for current MCPServerStdio |
| =job_discovery= | company search/create, job post create/find/search | Email pipeline agent |
| =scoring= | get_job_posts, get_career_data, score_job_post, get_scores | Scoring agent |
| =application_tracking= | job application CRUD, update_job_post, find_by_link | Status update agent |
| =scrape_management= | create_scrape, get_scrapes, update_scrape | Scrape queue agent |

Custom tool lists also accepted: =CareerCaddyToolset(scope=["score_job_post", "get_career_data"])=

*** Implementation: =ai/lib/toolsets.py= (new file)
Contains:
- =CareerCaddyDeps= dataclass
- =TOOL_REGISTRY= — dict mapping tool names to =api_tools.*= functions
- =SCOPES= — dict mapping scope names to sets of tool names
- =_make_tool_wrapper(fn)= — builds a =RunContext=-aware wrapper that preserves the original function's signature (minus the =api= param) for pydantic-ai schema generation
- =CareerCaddyToolset(AbstractToolset[CareerCaddyDeps])= — delegates to internal =FunctionToolset=
- Factory functions: =job_discovery_toolset()=, =scoring_toolset()=, etc.

*** Key challenge: signature preservation
Pydantic-ai uses =inspect.signature()= to build JSON schemas for tool parameters. The wrapper must copy the original function's parameter list (minus =api: ApiClient=) and construct a new =__signature__=. Same pattern FastMCP uses internally for =@server.tool()=.

*** Migration path
1. Create =ai/lib/toolsets.py=
2. Update =career_caddy_agent.py=: replace =MCPServerStdio= with =CareerCaddyToolset(scope="career_caddy")=, add =deps_type=CareerCaddyDeps=
3. Update =job_email_to_caddy.py=: indirect — it calls =add_job_post()= which uses the agent
4. =email_classifier_agent.py=: no change (uses email server only)
5. =job_extractor_agent.py=: no change (pure LLM, no tools)

*** Security: browser credentials are unreachable
- =lib/api_tools.py= imports only =httpx=, =json=, =pydantic= — zero knowledge of =secrets.yml= or browser automation
- =lib/toolsets.py= imports only from =lib/api_tools.py= and =pydantic_ai= — MUST NOT import from =lib/browser/*= or =mcp_servers/browser_server.py=
- =CareerCaddyDeps= holds only =api_token= + =base_url= — no credential paths
- No tool in the registry can read =secrets.yml=, trigger a browser login, or access notmuch email
- Browser/email capabilities live exclusively in =browser_server.py= and =email_server.py=, which are never imported by the toolset
- This is the same isolation model as =public_server.py= — enforced by import boundaries, not runtime checks

*** What stays the same
- =career_caddy_server.py= / =public_server.py= / =gateway.py= — still serve external MCP clients
- =lib/api_tools.py= — the single source of truth for tool implementations
- Email and browser MCP servers — separate tool sources, outside this scope

** MCP deployment at mcp.careercaddy.online
:PROPERTIES:
:CREATED:  [2026-04-10]

*** Architecture decision
Separate =public_server.py= — NOT the existing gateway. The gateway aggregates email + browser + caddy tools for local use. The public server exposes only career-data tools (21 total) behind per-client API key auth.

#+BEGIN_SRC
MCP Client → mcp.careercaddy.online (Caddy, TLS) → public_server.py (FastMCP) → Django API → PostgreSQL
#+END_SRC

*** Auth: API key pass-through
- FastMCP's =TokenVerifier= base class validates =jh_*= API keys by calling =GET /api/v1/me/= on the Django API.
- On success, the raw token is stored in =AccessToken.token= and forwarded on every downstream HTTP call.
- The Django API resolves user identity from the key — the MCP server never touches the DB.
- Each tool reads the token via =get_access_token()= from FastMCP's dependency injection.
- =get_current_user= tool: returns the authenticated user's profile so agents know who they act as.

*** Future: Slack integration (aspirational)
- Prototype branch at =~/Projects/career_caddy_ai/lib/tiger-agents-for-work= can chat in Slack, but it's a hack on a fork of the tiger project — goal is to not depend on that fork.
- Multi-tenant requirement: the agent must act on behalf of whichever user is talking to it, keyed by that user's =jh_*= API key.
- Authorization must flow through: leveraging toolsets, some users should be read-only while others can write — MCP/toolset scoping, not hard-coded in the agent.
- Verification target: rework =ai/run_email_web.py= into a minimal test harness that exercises =mcp.careercaddy.online= end to end.

*** Tools exposed (21)
14 ported from =career_caddy_server.py=:
- Companies: create, find_by_name, search, get
- Job posts: create_with_company_check, find_by_link, search, get, update
- Applications: create, get, get_for_job_post, update
- Career data: get

7 new:

- =create_scrape= — queue URL with =status="hold"= for later processing
- =get_scrapes=, =update_scrape= — scrape CRUD

*** "hold" status for scrapes
API change: =ScrapeViewSet.create= accepts =status="hold"= and skips the =SCRAPING_ENABLED= check and browser dispatch. Creates the record and returns 201 immediately. Agents use this to queue URLs for a separate scraping process.

*** Security invariants
- =public_server.py= does NOT import =email_server=, =browser_server=, =gateway=, or =lib/browser/*=
- Docker service does NOT mount =secrets.yml= or mail directories
- No =CC_API_TOKEN= env var on the container — all auth from clients

*** Gotcha: =invalid_token= debugging playbook (two shapes) :IMPORTANT:
Every time the MCP silently returns =invalid_token= for a token that works
against the public API, the cause has been a non-200 on the internal =mcp →
http://api:8000/api/v1/me/= hop. FastMCP's =verify_token= returns =None= on
anything other than 200, so the failure is invisible until you reproduce the
upstream call from inside the mcp container.

**** Shape 1: =ALLOWED_HOSTS= rejection (400 DisallowedHost) — fixed 2026-04-17
Django's =ALLOWED_HOSTS= did not include the Docker service hostname =api=.
Internal requests sent =Host: api:8000=, Django responded 400 DisallowedHost
before auth ran. Outer =api/settings.py= was dead code (never used by
=DJANGO_SETTINGS_MODULE=job_hunting.settings=), so its attempt to append
="api"= to the list never took effect in prod.

Fix (commit a303170): unconditionally add ="api"= to =ALLOWED_HOSTS= in
=api/job_hunting/settings.py= and delete the dead outer =api/settings.py=.

**** Shape 2: =SECURE_SSL_REDIRECT= 301 — fixed 2026-04-17
Django enables =SECURE_SSL_REDIRECT=True= in prod
([[file:api/job_hunting/settings.py::SECURE_SSL_REDIRECT][settings.py:293]]). The
Caddy edge sets =X-Forwarded-Proto: https= so public requests pass, but
=mcp → http://api:8000= internal calls don't go through Caddy. Without the
header, Django responds =301 → https://api:8000/= and the MCP verifier sees
a non-200 → =invalid_token=.

Fix (ai commit 58c1f3d): every internal httpx call to the API includes
=X-Forwarded-Proto: https=. Callers that send it today:
- =ai/lib/api_tools.py= =ApiClient._headers= (and =post_file=, fixed 2026-04-17)
- =ai/lib/models/career_caddy.py= =APIContext=
- =ai/lib/usage_reporter.py=
- =ai/mcp_servers/chat_server.py= =_fetch_user_profile=
- =ai/mcp_servers/public_server.py= =ApiKeyTokenVerifier.verify_token= (fixed 2026-04-17)

**** Diagnostic playbook
When =curl https://mcp.careercaddy.online/mcp= returns 401 invalid_token but
the same token returns 200 from =curl https://api.careercaddy.online/api/v1/me/=:

1. On the VPS, reproduce the exact internal call that =verify_token= makes:
   #+begin_src sh
   docker compose -f docker-compose.prod.yml exec -e TOKEN='jh_...' mcp \
     python -c 'import os,httpx;r=httpx.get(os.environ["CC_API_BASE_URL"]+"/api/v1/me/",\
     headers={"Authorization":"Bearer "+os.environ["TOKEN"]});\
     print(r.status_code);print(dict(r.headers));print(r.text[:500])'
   #+end_src
2. Read the status code:
   - *400 DisallowedHost* → shape 1, rebuild api image with =api= in =ALLOWED_HOSTS=.
   - *301 → location: https://api:8000/...* → shape 2, add =X-Forwarded-Proto: https= to the caller.
   - *401* → token actually revoked or wrong user; regenerate via =/api/v1/api-keys/=.
   - *200* → mcp-side bug; check the new =logger.warning= in =verify_token=.
3. If the container shows =unhealthy= status that's a red herring — the healthcheck hits =/mcp= which always 401s unauth. The service can still be functioning.

**** Startup probe
=public_server.py:_probe_upstream_api()= runs =GET /api/v1/healthcheck/= with
=X-Forwarded-Proto: https= at boot and exits non-zero on any non-200 or
connect error. This turns future misconfigs (ALLOWED_HOSTS drift, wrong
=CC_API_BASE_URL=, downed api container) into a loud crashloop with a
clear log line, rather than a quietly-up container that rejects every token.

*** Agent identity awareness
:PROPERTIES:
:CREATED:  [2026-04-11]
:END:
Two paths for the agent to know the user's identity:

**** Web chat (chat_server.py) — JWT pass-through (Option C)
- Frontend sends JWT in =Authorization: Bearer= header to =/api/v1/chat/=
- Django proxy validates the JWT, then forwards the *same token* to the chat service
- No temporary API keys created — eliminates the DB write per message overhead
- Chat service calls =GET /api/v1/me/= with the forwarded token to fetch user profile
- Profile (name, email, linkedin, github) is injected into the system prompt
- Same token used for all downstream tool calls via =ApiClient=
- System prompt explicitly instructs the agent to use this data and never deny access
- Name fallback chain: =first_name last_name= → =username= → =email= → "authenticated but no profile"
- Chat service accepts both JWTs and =jh_*= API keys — the API's auth stack handles either

**** Public MCP server (public_server.py)
- FastMCP =instructions= parameter tells MCP clients to call =get_current_user= at session start
- =get_current_user= tool returns the authenticated user's full profile from the token claims
- No system prompt injection needed — the MCP client's own LLM handles the instructions
- Auth: =jh_*= API keys provisioned by the user (long-lived, for CLI/Slack/third-party apps)

**** Key insight
The web chat injects profile into the system prompt (server-side). The MCP server provides
instructions for the client to fetch it (client-side). Both paths achieve the same result:
the agent knows who the user is from the first message.

Two auth token types serve different clients:
- JWT: browser sessions (short-lived, auto-refreshed by the SPA)
- API key (=jh_*=): MCP clients, Slack, CLI, third-party apps (long-lived, user-provisioned)

*** Running lint
#+begin_src bash
npm --prefix /path/to/frontend run lint        # all four in parallel
npm --prefix /path/to/frontend run lint:js     # just ESLint
npm --prefix /path/to/frontend run lint:hbs    # just template lint
npm --prefix /path/to/frontend run lint:format # just prettier check
#+end_src

*** Fixing prettier issues
Prettier must be run from within the frontend submodule context so that =prettier-plugin-ember-template-tag= resolves correctly. Running prettier from the repo root will fail because the root =noop.js= can't find the plugin.

#+begin_src bash
# Check what's wrong:
npm --prefix /path/to/frontend run lint:format

# Auto-fix (pass files after -- --write):
npm --prefix /path/to/frontend run lint:format -- --write /absolute/path/to/file.hbs
#+end_src

*** Key gotcha
Do NOT run =npx prettier= from the repo root — it picks up the root config and fails to resolve the Ember template tag plugin. Always use =npm --prefix frontend run lint:format=.

#+end_src

* Color Palette System
:PROPERTIES:
:CREATED:  [2026-04-11]

** Architecture
Two independent dimensions: *mode* (light/dark) and *palette* (indigo/blue/jade/rose/amber/violet).
Applied via =data-theme= and =data-palette= attributes on =<html>=.
ThemeService stores both in localStorage (=theme-mode=, =theme-palette=).
Settings > Appearance tab lets users pick palette + toggle dark mode.

** How to add a new palette

1. Pick a color family (Tailwind reference or custom). You need shades 50-900.
2. Convert hex to space-separated RGB: =printf "%d %d %d\n" 0xRR 0xGG 0xBB=
3. Add a CSS block in =frontend/app/styles/app.css= after existing palette blocks:
   #+begin_src css
   [data-palette="teal"] {
     --accent-50:  240 253 250;
     --accent-100: 204 251 241;
     /* ... shades 200-800 ... */
     --accent-900: 19 78 74;
     --accent: rgb(var(--accent-500));
     --tee-bg: rgb(var(--accent-200));
     --page-bg: #f0fdfa;
   }
   #+end_src
4. Add the name to =PALETTES= array in =frontend/app/services/theme.js=.
5. Add swatch colors to =SWATCH_COLORS= in =frontend/app/controllers/settings/appearance.js=.
6. No template changes needed — all templates use =accent-*= classes.
7. Test in both light and dark modes.

* AI Chat — Architecture Reference
:PROPERTIES:
:CREATED:  [2026-04-13]
:END:
New agents: read this section to understand how AI chat works end-to-end.

** Overview
Chat is a three-layer streaming system: Ember frontend → Django API proxy → Pydantic-AI chat server.
The user types a message, it flows through all three layers, and the response streams back as SSE events.
The AI agent has access to 27 tools that CRUD the user's career data via the same API.

** Message flow (end-to-end)
#+begin_src
User types in frontend (Chat::Panel textarea)
  ↓
ChatService.sendMessage(text)                    [frontend/app/services/chat.js]
  - adds user message + empty assistant placeholder to @tracked messages array
  - calls session.ensureFreshToken(90) to ensure valid JWT
  - POST ${baseUrl}chat/ with JSON body:
    {message, history, conversation_id, page_context}
    Authorization: Bearer <jwt>
  ↓
chat_proxy(request)                              [api/job_hunting/api/chat.py]
  - validates JWT via DRF JWTAuthentication
  - extracts raw token string from Authorization header
  - builds payload: {message, token, history, conversation_id, page_context}
  - streams POST to http://localhost:8031/chat via httpx.Client.stream()
  - returns StreamingHttpResponse (text/event-stream)
  ↓
chat(request)                                    [ai/mcp_servers/chat_server.py]
  - _fetch_user_profile(token): GET /api/v1/me/ → name, email, linkedin, github
  - _build_agent(user_profile, page_context): creates pydantic-ai Agent
    with dynamic system prompt (profile + page context injected)
  - CareerCaddyDeps(api_token=token, base_url) passed to agent.run_stream()
  - agent streams text chunks + makes tool calls as needed
  - each tool uses the same JWT for API calls → data is scoped to the user
  ↓
SSE event stream yields back through all layers:
  data: {"type": "text", "content": "chunk..."}       ← partial text
  data: {"type": "navigate", "url": "/resumes"}       ← programmatic navigation
  data: {"type": "done", "content": "full text",      ← final response
         "conversation_id": "uuid", "usage": {...}}
  data: {"type": "error", "content": "error msg"}     ← on failure
  ↓
Frontend SSE reader accumulates text chunks, updates last message,
handles navigate events, stores conversation_id for multi-turn.
#+end_src

2. Server emits ={type: "navigate", url: "/path"}= SSE event. Frontend handles it in the SSE reader.

=main-application.js= sets =chat.currentPage = {route, url, params}= on every route change.
Sent in the chat POST body. Chat server injects "Current Page" section into system prompt.
Enables the AI to understand "this job post" or check data on the current page.

** Motivation
The browser scraper agent is a pass-through that wastes LLM tokens — the LLM just calls =scrape_page(url)= and echoes the result. By calling =scrape_page()= directly as a Python function and only using LLMs for the intelligent steps (job extraction via Sonnet, CRUD via gpt-4o-mini), we save ~56% per scrape.

Browser scraping on credentialed sites fails silently in prod Docker (no sessions/cookies). The hold-scrape pattern lets admin queue URLs from the prod UI, then a local poller processes them with a headed browser and saved sessions.

** Architecture

*** Multi-model pipeline (cost-optimized)
| Step                  | Model              | Cost     |
|-----------------------+--------------------+----------|
| =scrape_page()=       | None (direct call) | $0.00    |
| =extract_job_from_content()= | Claude Sonnet      | ~$0.10   |
| =add_job_post()=      | gpt-4o-mini        | ~$0.002  |
| *Total per scrape*    |                    | *~$0.10* |

*** Hold poller — scrape-only worker

*What it is:* A local process (not Docker) that polls the Career Caddy API for
scrapes with =status=hold=, runs Camoufox to fetch the page, and PATCHes the
content back. The API handles everything downstream (extraction, job post
creation, scrape profile updates).

*Why it exists:* Browser scraping is too heavy for the VPS. The poller runs on
a desktop or Pi, keeping the VPS as a thin API + frontend layer.

*Running it:*
| Method | Command |
|--------+-----------------------------|
| Standalone (prod) | =cd ai && cp .env.prod.example .env && uv run caddy-poller= |
| Standalone (local dev) | =cd ai && CC_API_BASE_URL=http://localhost:8000 CC_API_TOKEN=xxx uv run caddy-poller= |
| With dev stack | =make up-full= (Docker stack + poller in background) |

*Prerequisites:*
- =uv sync= + =uv run python -m camoufox fetch= (one-time)
- =CC_API_BASE_URL= and =CC_API_TOKEN= env vars
- No LLM keys needed on the worker — all LLM work happens API-side

*Backoff:* The poller uses exponential backoff: 2, 4, 8, 16, 32, 64, 128, 256
seconds. Resets to 2s when work is found. Stops after the full sequence with no
work (~510s total). This means you run it, it processes any hold scrapes, backs
off, and exits. Run it again when you queue more holds.

*Flow:*
#+BEGIN_SRC
caddy-poller starts
  poll: GET /api/v1/scrapes/?filter[status]=hold
  found work?
    yes → for each hold scrape:
            PATCH status → running (note: "Poller picked up")

* Domain Scrape Profiles — learning loop for scrape cost reduction
:PROPERTIES:
:CREATED:  [2026-04-14]
:END:
[[file:todo.org::*Domain scrape profiles][↑ todo.org]]

** Motivation
The job extractor LLM call (~$0.025/scrape with Haiku) is the main per-scrape cost. For domains scraped frequently (LinkedIn, Indeed, Glassdoor), the LLM re-discovers the same page structure every time. A domain-level profile that captures what worked before could eliminate or reduce the LLM cost on repeat scrapes.

** Model: =ScrapeProfile= (per hostname)
| Field                | Type     | Purpose                                              |
|----------------------+----------+------------------------------------------------------|
| =hostname=           | str (PK) | Normalized domain (e.g. =linkedin.com=)              |
| =requires_auth=      | bool     | Learned from =login_wall_detected= failures          |
| =avg_content_length= | int      | Baseline to detect anomalies (too short = blocked)   |
| =success_rate=       | float    | Track reliability over time                          |
| =css_selectors=      | json     | Selectors that work for this domain's job pages      |
| =extraction_hints=   | text     | Free-text LLM context: "salary in EUR", etc.         |
| =page_structure=     | text     | LLM-generated description of the page layout         |
| =last_success_at=    | datetime | Stale profile detection                              |
| =scrape_count=       | int      | Usage tracking                                       |
| =created_by=         | FK       | Multi-tenancy                                        |

** How the learning loop works
1. First scrape of =linkedin.com= → no profile → full LLM extraction ($0.025)
2. Extraction succeeds → auto-populate profile from the result
   - LLM writes =extraction_hints=: "title in =.top-card-layout__title=, company in =.topcard__org-name-link=, salary format USD annual"
   - =avg_content_length= set from this scrape
   - =requires_auth= set from ScrapeStatus history
3. Next =linkedin.com= scrape → profile hints injected into extractor prompt → cheaper/faster
4. If extraction fails or quality degrades → fall back to full LLM, flag profile as stale
5. Staff can review/edit profiles in admin UI

** Tiered scrape cost structure
| Tier | When                          | Model          | Cost/scrape |
|------+-------------------------------+----------------+-------------|
|    0 | Known domain + CSS selectors  | None (determ.) | $0.00       |
|    1 | Known domain + hints          | gpt-4o-mini    | ~$0.005     |
|    2 | Unknown domain                | Haiku          | ~$0.025     |
|    3 | Complex / high-value page     | Sonnet         | ~$0.10      |

Tier selection can be automatic (profile exists? use it) or manual (admin chooses tier per scrape).

** How extraction_hints get written
After a successful extraction, ask the LLM one follow-up question:
#+begin_src

in 2-3 sentences so a cheaper model can extract from similar pages next time.
Include: where the title, company, salary, and description appear on the page."
#+end_src
This is a one-time ~$0.01 cost that pays off over many future scrapes.

* Builder-form style conventions                                    :IMPORTANT:

** Layout classes
| Class | Behaviour | Use for |
|-------+-----------+---------|
| =builder-form= | 2-col grid: =max-content 1fr= | Default form layout (label + input per row) |
| =builder-form.two-col= | Equal 2-col grid, fields side-by-side | Settings profile, questions form |
| =.field= | Direct child: =display:contents= (dissolves into grid) | Standard label+input pair |
| =.field-full= | =flex-direction:column=, spans both cols. *All* child =input/textarea= get =width:100%= | Full-width fields (address, description, question content) |

** Key gotcha: =.field-full= forces =width:100%= on all descendant inputs
If you put a grid or flex row of multiple inputs inside =.field-full=, each input expands to 100% and stacks vertically. Use =.field-row= instead when you need multiple inputs on one line (e.g. the links editor on settings/profile).

** Inline styles
Never use inline =style="..."= attributes. Tailwind utilities are the standard. If a Tailwind class is being overridden by =builder-form= CSS, choose the correct form layout class (=field-row= vs =field-full=) rather than fighting it with inline styles.

** Read-only views
Match the edit form's column layout. Use =grid grid-cols-2 gap-x-6 gap-y-4= on the =<dl>=. Full-width items (address, links) get =col-span-2=.

* Resume serializer — JSON:API relationship linkage + sideloading

** What was wrong
=ResumeSerializer.to_resource= embedded sub-resources (summaries, certifications,
educations, experiences, skills, projects) as plain JSON arrays in =attributes=.
It declared them as =relationships= with =links= but *no =data= linkage*.

Ember Data ignored the embedded attributes and tried to fetch via =related= URLs
(e.g. =/api/v1/resumes/30/certifications=) — which don't exist as nested endpoints.
Result: 404s and missing data (summaries carousel, certifications display).

** What was fixed

*** ResumeSerializer (=api/job_hunting/api/serializers.py=)
Removed ~130 lines of manual attribute embedding. Added proper JSON:API =data= linkage
arrays using the existing =get_related()= method. Pattern matches =DjangoUserSerializer=.
The =summary= convenience attribute (active summary text string) is kept.

*** ExperienceSerializer (same file)
Added =data= linkage for =descriptions= relationship. Without this, Ember Data couldn't
match description records in =included= to their parent experience.

*** ResumeViewSet (=api/job_hunting/api/views.py=)
Added =_default_includes= class attribute. Both =list()= and =retrieve()= auto-include
all sub-resources when no =?include== param is specified. No extra frontend fetches needed.

*** _build_included ownership filter (=views.py=)
Resources with =user_id=None= were silently dropped by the ownership filter
(=None != user_id= is =True=). Fixed: skip the filter when =user_id is None= — if
reached through a relationship on an already-authorized parent, they're safe to include.

*** Summaries carousel (=frontend/app/components/summaries/list.js=)
- Route loads all summaries via =store.findAll('summary')=
- Controller passes =store.peekAll('summary')= (all user summaries, not just resume-linked)
- =_navigate()= now PATCHes the summary with =active=true= + resume relationship
  (was incorrectly pushing into resume's hasMany and saving the resume)
- Position indicator added (e.g. "3 / 12")

** Key pattern: JSON:API relationship =data= linkage
When a serializer declares to-many =relationships=, =BaseSerializer.to_resource= only emits
=links= (no =data=). This is correct for relationships that are paginated or expensive.
For relationships that should resolve from =included= sideloads, the serializer's
=to_resource= override must add =data= arrays:

#+BEGIN_SRC python
# In the serializer's to_resource():
for rel_name, rel_type in rel_defs:
    _, items = self.get_related(obj, rel_name)
    linkage = [{"type": rel_type, "id": str(item.id)} for item in items]
    res.setdefault("relationships", {})[rel_name] = {
        "data": linkage,
        "links": existing_links,
    }
#+END_SRC

Without =data=, Ember Data has no way to match =included= records to their parent —
even if the records are in the payload.

** Key pattern: async hasMany proxy resolution
Even with proper sideloading, Ember Data's =async: true= hasMany returns a =PromiseManyArray=.
Accessing =.length= before resolution returns 0. Routes must =await= relationships:

#+BEGIN_SRC javascript
const resume = await this.store.findRecord('resume', resume_id);
await Promise.all([resume.skills, resume.summaries, resume.projects]);
// Now .length and .objectAt() work synchronously in templates
#+END_SRC

This await resolves instantly from the store (no extra fetch) when data is sideloaded.

* Full-height page layout (caddy chat)
:PROPERTIES:
:CREATED:  [2026-04-15]
:END:

** Problem
The =/caddy= full-page chat didn't fill the viewport. Input sat in the middle of the page
instead of at the bottom. The empty space below was unusable.

** Root cause: =overflow-y: auto= breaks flex height constraint
=.course-main= has =overflow-y: auto= by default (so normal pages scroll). When we add
=display: flex; flex-direction: column= for the caddy-fullpage override, =overflow: auto=
tells the browser the flex container can grow and scroll — so flex children don't get
constrained to the available height. They render at their natural size and the container
scrolls. Since the chat panel is smaller than the viewport, it just sits in the middle.

** Fix
=.course-main:has(.caddy-fullpage)= now sets =overflow: hidden=. This forces the flex
children to fit within the grid cell's actual height. The chat panel's own messages div
has =overflow-y: auto= so messages still scroll internally.

** Height propagation chain (all must be flex/grid with constrained overflow)
#+BEGIN_SRC

  .course-main   → grid-area: main; overflow: hidden (was auto — that broke it)
    :has(.caddy-fullpage) → display: flex; flex-direction: column
      #subnav-portal → auto height (breadcrumbs bar)
      .fairway       → flex: 1; min-height: 0; display: flex; flex-direction: column
        .caddy-fullpage → flex: 1; min-height: 0; display: flex; flex-direction: column
          Chat::Panel   → flex flex-col h-full
            header      → auto
            #chat-messages → flex-1 overflow-y-auto (this scrolls)

*** How AnswerService works (the reference implementation)

The answer generation pipeline:

#+BEGIN_SRC
Frontend POST /api/v1/answers/ { ai_assist: true, injected_prompt: "..." }
  → AnswerViewSet.create (views.py ~5940)
    → extracts injected_prompt from request payload
    → builds career_markdown from /career-data/ via ApplicationPromptBuilder.build_from_career_data()
    → spawns async thread calling AnswerService.generate_answer()
      → load_context_for_question(): resolves user, application, job_post, company, resumes, cover_letters, Q&A history
      → ApplicationPromptBuilder.build(context, injected_prompt=..., career_markdown=...)
        → assembles prompt in this order:
           1. Instructions (preamble — system rules)
           2. ## Additional Instructions (injected_prompt — caller-supplied directives)
           3. ## Question to Answer (the actual question)
           4. ## Career Profile (career_markdown, if provided — replaces resumes)
           5. ## Job Details
           6. ## Company
           7. ## Resume(s) Summary
           8. ## Cover Letters
           9. ## Q&A History
      → write_prompt_to_file() (debug only — writes to api/runtime/prompts/)
      → _call_ai(prompt) → OpenAI chat.completions
      → saves Answer record
#+END_SRC

The prompt is structured so =head= on any logged prompt file shows: what rules apply, what's being asked, and what special instructions were given — before the bulk of supporting context.

* Lightweight JobPost from email + on-demand scrape
[2026-04-17 api+frontend] [[file:~/.claude/plans/hazy-herding-river.md][Plan]]

** Motivation
Reading email and POST-scraping every job link was too much overhead for
pages the user would never open. Emails usually carry only the *title +
link*, so pre-scraping wastes browser time and infra. Better: drop a
stub JobPost fast, let the user browse, scrape on demand.

** Flow
- =career_caddy_automation= POSTs minimal JobPost ={title, link}= to
  =/api/v1/job-posts/= (company nullable, description omitted).
- When email *does* carry company info, the existing
  =create_job_post_with_company_check= path in the automation repo is
  unchanged.
- User browses =/job-posts= → sees a *needs scrape* badge on any post
  with no description.
- User opens one → clicks *Run scrape* on =job-posts.show=. Creates a
  Scrape with =status="hold"= bound to the JobPost (and its Company if
  present). Hold-poller picks it up; extractor fills in the rest.

** Decisions
- *Null company, no sentinel.* JobPost.company FK is already nullable
  and the list template already renders an amber "missing" badge.
  Seeding an "Unknown" Company would just bloat the company pool.
- *"Scraped" = has description.* The indicator is description-based,
  not Scrape-row-based — a failed/hold Scrape record doesn't mean the
  post is usefully populated, and an imported post may have a
  description without a Scrape. =JobPost.needsScrape= is a getter,
  not a sideloaded relationship — no =include=scrapes= on the index.
- *"hold" not "pending" for on-demand scrapes.* =pending= sits there
  doing nothing; the dispatcher does not auto-pick it. =hold= is the
  status the hold-poller watches. Match the existing chat-scrape
  convention.
- *Idempotent JobPost POST.* Duplicate-link now returns the existing
  post with 200 instead of 409 (reverses the earlier Inbox decision
  ="when POSTing a new job-post, if the url is already there don't
  500"=). Automation can safely retry without dedup logic of its own.
- *Explicit =relationships.job-post= on scrape POST wins over URL
  match.* URL-match fallback stays as a safety net. Company is
  inherited from the linked post when not supplied.

** Files
- =api/job_hunting/api/views/jobs.py= — duplicate-link 200 return
- =api/job_hunting/api/views/scrapes.py= — explicit job_post binding
  in the hold branch
- =frontend/app/models/job-post.js= — =needsScrape= getter
- =frontend/app/components/job-posts/list.hbs= — badge
- =frontend/app/templates/job-posts/show.hbs= + controllers/show.js —

=openai:gpt-4o-mini= the big prompt caused: false refusals to share the

Fixes landed:
1. *Model swap* — =CHAT_MODEL=anthropic:claude-haiku-4-5= (our own

2. *Agent delegation (pydantic-ai agent-as-tool pattern)* —

   own tight system prompt + 5-tool scope. Main chat exposes a single
   =ask_onboarding_wizard(user_message)= tool
   (=lib.toolsets.onboarding_delegation_toolset=) and returns the
   sub-agent's output verbatim (preserving navigate markers).
3. *Scope split* — new =main_chat= scope excludes
   =reconcile_onboarding=, =edit_profile_onboarding=,
   =import_resume_from_url=. Sub-agent owns those. This forces the main
   agent to delegate for onboarding instead of doing it itself badly.
4. *Per-turn context via deps* — =CareerCaddyDeps= now carries
   =user_profile=, =onboarding=, =page_context=. The delegation tool

5. *User-profile rendering* — =_fetch_user_profile= now emits every

6. *Cost-saving reconcile rule* — sub-agent only calls
   =reconcile_onboarding= when a snapshot =false= key is contradicted

** Tool-call visibility in chat (staff-only, first step toward AG-UI)
[2026-04-18 ai+frontend]

*** Why
Step B fixed onboarding delegation but haiku still occasionally emits
"I'll check X" text without calling a tool. The user has no signal
that the agent is actually doing anything vs. just talking. Our SSE
was text-only; tool calls happened silently server-side.

*** Reference — pydantic-ai AG-UI integration
pydantic-ai ships an AG-UI adapter that streams a richer SSE event
stream with explicit tool-call events. Their chat UI example renders
each tool invocation as a visible step so users (and developers) can
see exactly what the agent did.

Docs we consulted:
- [[https://ai.pydantic.dev/ui/ag-ui/][Pydantic AI AG-UI integration]]
- [[https://ai.pydantic.dev/examples/ag-ui/][Pydantic AI AG-UI example]]
- [[https://docs.ag-ui.com/concepts/events][AG-UI event types reference]]
- [[https://docs.ag-ui.com/introduction][AG-UI protocol overview]]
- [[https://ai.pydantic.dev/agent/][Pydantic AI agent.iter() / node.stream() API]]

AG-UI event types relevant to us (shapes at
=docs.ag-ui.com/concepts/events=):
- Lifecycle: =RunStarted=, =RunFinished=, =RunError=, =StepStarted=,
  =StepFinished=
- Text: =TextMessageStart=, =TextMessageContent=, =TextMessageEnd=,
  =TextMessageChunk=
- Tools: =ToolCallStart=, =ToolCallArgs=, =ToolCallEnd=,
  =ToolCallResult=, =ToolCallChunk=
- State: =StateSnapshot=, =StateDelta=, =MessagesSnapshot=
- Reasoning: =ReasoningStart=, =ReasoningMessageContent=, …

Each event has a correlation id (=toolCallId=, =messageId=) so the
frontend can update in-place as chunks arrive.

*** What we shipped (incremental)
Not a full AG-UI adoption — just two new event types on our existing
handcrafted SSE so staff can see tool activity. Matches pydantic-ai's
intent without forcing a frontend migration.

Server (=ai/mcp_servers/chat_server.py=): switched =_stream_once= from
=agent.run_stream().stream_text()= to =agent.iter()= +
=node.stream()=. Walks the node graph:
- =Agent.is_model_request_node(node)= → iterate =PartDeltaEvent=,
  =PartStartEvent=. =TextPartDelta= becomes =text= SSE. Track
  =tool_call_id= → =tool_name= from =PartStartEvent= (some models
  carry the name there before the tool node).
- =Agent.is_call_tools_node(node)= → iterate =FunctionToolCallEvent=
  (emit =tool_call_start= with id/name/args) and
  =FunctionToolResultEvent= (emit =tool_call_end= with id/name/
  truncated content, also fire =reload= here if the tool maps to a
  reloadable resource — earlier we ran the reload pass after the
  stream, now it's inline right after the tool succeeds).

=all_messages()= / =new_messages()= / =usage()= are read from
=run.result= after the =async for node in run:= loop exits.

Frontend:
- =app/services/chat.js= — new SSE cases =tool_call_start= (appends to
  the current assistant message's =toolCalls= array) and
  =tool_call_end= (finds by id, marks done, attaches truncated
  result). New helpers =_updateLastMessage=, =_appendToolCall=,
  =_finishToolCall=.
- =app/components/chat/message.{js,hbs}= — renders a tool-call
  breadcrumb strip above the assistant bubble, gated by
  =currentUser.user.isStaff=. Each call shows a pulsing dot (pending)
  or check mark (done) + the tool name.

*** Why staff-only for now
User wants to validate the UX as a staff member before rolling out.
If it reads as useful rather than noisy, we flip the gate and show
it to everyone. Flipping is a one-line change in
=chat/message.js::showToolCalls=.

* Scrape → Parse → JobPost workflow (post-refactor)                 :IMPORTANT:
As of [2026-04-19], the parse pipeline has a `force` flag that unifies
the three user-initiated re-parse paths with the original first-pass
pipeline. This entry is the single reference for how a Scrape turns into
a JobPost and when fields get overwritten.

** Actors
- =Scrape= — raw capture (html and/or job_content). Has =status= column
  (=pending|hold|running|authenticating|login_failed|scraping|extracting|completed|failed|success=)
  and an append-only =scrape_statuses= audit log via =_log_scrape_status=.
- =JobPost= — structured extraction. =link= is unique; =company= FK is
  preserved once set (we never reassign on re-parse).
- =parse_scrape(scrape_id, user_id=None, sync=False, force=False)= —
  single entry point in =api/job_hunting/lib/parsers/job_post_extractor.py=.
  Default: spawns a daemon thread; if scrape already has =job_post_id=
  and =force=False=, bails out (idempotency). =force=True= always runs.
- =JobPostExtractor.parse(scrape, user, force)= — Tier 0 (CSS selectors
  from ScrapeProfile) first, then =analyze_with_ai= (LLM). Returns True
  on success.
- =process_evaluation(..., force)= — the merge policy:
  - =force=True AND scrape.job_post_id set= → fetch THAT post, overwrite
    non-None extracted fields (title, description, salary_min/max,
    posted_date, extraction_date, location, remote). =company= left
    alone. This is the re-parse path.
  - otherwise: look up JobPost by =scrape.url= (unique link), else
    get_or_create by (title, company). Fill only previously-None fields.

** Entry points (how a Scrape becomes a parse job)
| Trigger                                | Force? | Typical path                                                       |
|----------------------------------------+--------+--------------------------------------------------------------------|
| Hold-poller writes =status=completed=    | depends | =ScrapeViewSet.partial_update= → =_maybe_trigger_extraction(pk, force=_reparse_force(pk))=. =_reparse_force= returns True iff =scrape.job_post_id= is already set (→ user re-ran a scrape on an existing post). |
| =/scrapes/from-text/=                    | False  | Creates =Scrape(status=pending)=, calls =parse_scrape(sync=False)=. First-pass flow: find-or-create the JobPost. |
| =/scrapes/{id}/parse/= (manual button)   | *True* | User said "do it now"; always overwrite with fresh extraction.     |
| =/job-posts/{id}/reextract/= (paste)     | *True* | Creates =Scrape(status=pending, job_post=obj)=, calls =parse_scrape(force=True)=. Merges into the existing post. |
| =/job-posts/{id}/show= Run scrape button | via poller | Creates =Scrape(status=hold, job_post=obj)=. Poller picks it up; when it writes =completed=, =_reparse_force= sees =job_post_id= set → =force=True=. |

** Status lifecycle inside =parse_scrape._run=
1. =pending= (or whatever status the caller set before invoking).
2. =extracting= — logged as soon as =parse_scrape= starts real work.
3. =updating_profile= — ScrapeProfile success/failure counters tick.
4. =completed= OR =failed= — terminal. Frontend pollable stops.

** Frontend polling (=services/pollable.js=)
Three UI surfaces kick off a scrape and poll for terminal:
- =/job-posts/paste= controller — text-only paste; polls the from-text
  scrape; onComplete navigates to the new =/job-posts/{id}=.
- =/job-posts/:id/show::runScrape= — creates =Scrape(status=hold)= then
  =pollable.poll(scrape)= with spinner.begin; scrape reaches terminal,
  spinner ends, flash fires. =_reparse_force= above ensures a re-scrape
  of an already-linked post still merges fields.
- =/job-posts/:id/edit::reextractFromPaste= — POST to =/reextract/=
  returns the pending Scrape, frontend polls; onComplete reloads the
  JobPost via =store.findRecord(..., {reload: true})= so form fields
  refresh in place.

** Why the force flag exists (the bug it fixed)
Before [2026-04-19], =parse_scrape= early-bailed when =scrape.job_post_id=
was already set, and =process_evaluation= only filled previously-None
fields on merge. That broke two user stories:
1. Re-running a scrape from =/job-posts/show= — the poller wrote
   =completed= but the auto-parse did nothing because the scrape already
   had a =job_post_id= from its =Scrape.objects.create(jobPost=this.model)=
   call.
2. Manual Parse on =/scrapes/show= — the =parse= action called
   =parse_scrape= which bailed for the same reason; even when it didn't
   bail, the merge policy kept old descriptions around.
Both cases now route through =force=True= → overwrite non-None fresh
fields, preserve company.

** Known edge cases
- =_reparse_force= fires on ANY transition to =completed= on a linked
  scrape, including a poller re-run triggered by status edits from the
  UI. If that ever feels too aggressive, tighten the check to "status
  transitioned from non-terminal to =completed=" rather than "current
  status is =completed=".
- =title= is special: it's a =get_or_create= lookup key on the cold path
  so it doesn't live in =job_defaults=. The force branch handles it
  explicitly. Anyone adding new overwrite-on-force fields should put
  them in =job_defaults= AND check whether they're also used as lookup
  keys downstream.
- =_to_decimal_safe= helper in =views/jobs.py= was deleted with the
  old =_run_reextract_async=. If you reintroduce it, import
  =from decimal import Decimal, InvalidOperation= at the top of the
  module.

* Hold Poller — obstacle resolution + probation-based graduation
:PROPERTIES:
:CREATED:  [2026-04-19]
:END:

Extends [[*Hold Poller, Screenshots, Multi-Model Pipeline][Hold Poller, Screenshots, Multi-Model Pipeline]] and
[[*Domain Scrape Profiles — learning loop for scrape cost reduction][Domain Scrape Profiles — learning loop for scrape cost reduction]].

** Motivation
The hold poller routinely hit =login_wall_detected= on LinkedIn URLs that
were actually authenticated — the page was just a "Welcome Back / Continue
as Doug" account chooser, or a "See more" button was truncating the job
body. Multiple failure modes that are trivial for a human (or vision model)
but opaque to simple text detection.

Approach: keep cheap/deterministic paths fast, reserve LLM calls for
genuinely novel obstacles, and *graduate* repeat obstacles into
deterministic selectors via a probation gate so the LLM only pays for
first-contact.

** Data home — all new state lives in =ScrapeProfile.css_selectors=
No migration. =css_selectors= is a =JSONField=; top-level keys carry
per-host config in addition to the original job_data selectors.

| Key                           | Shape           | Purpose                                                     |
|-------------------------------+-----------------+-------------------------------------------------------------|
| =job_data=                    | ={field: sel}=  | Original — discovered selectors for title/desc/etc.         |
| =ready_selector=              | string          | Wait-for-selector after =domcontentloaded= (SPA settle).    |
| =post_nav_wait_ms=            | int (≤30000)    | Fixed sleep after navigation when no ready_selector is set. |
| =interaction_hints=           | free-text       | Obstacle-agent playbook for this host.                      |
| =obstacle_click_selector=     | string          | Graduated selector that clears the known obstacle.          |
| =_ready_selector_candidate=   | ={sel, matches}= | Probation state for ready_selector.                         |
| =_obstacle_click_candidate=   | ={sel, matches}= | Probation state for obstacle_click_selector.                |

** Pipeline (chronological) — what fires in what order
In =mcp_servers/browser_server.py= =scrape_page()=:

1. =page.goto(..., wait_until="domcontentloaded")=
2. =asyncio.sleep(1)= + =wait_for_load_state("networkidle", timeout=15s)=
3. *Post-nav settle* (new):
   - if =ready_selector= is set → =wait_for_selector(timeout=10s)=
   - elif =post_nav_wait_ms= is set → =asyncio.sleep(min(ms, 30000)/1000)=
   - else → default =asyncio.sleep(4)=
4. *Expand truncations* (new): =_try_expand_truncations(page)=
   clicks any "See more / Show more / Read more" button whose *own text*
   contains "more". Selectors tried: =button.jobs-description__footer-button=,
   =button.show-more-less-html__button=, =button[aria-label*='see more' i]=,
   =button[aria-label*='show more' i]=, =:has-text= fallbacks.
5. Read body text, run 5× =_is_still_loading= loop.
6. Screenshot → SCREENSHOT_DIR.
7. Profile-selector check: blocked / authenticated / job_data matched.
   - blocked → return =blocked_page_detected=.
8. *Login-wall handling* — tiered:
   a. =_detect_login_wall(content)= fires on <200-word pages with
      strong signals (="sign in to discover"=, ="logging you in"=, etc.) or
      ≥2 weak signals (="sign in"=, ="welcome back"=, ="continue as"=, etc.).
   b. If "continue as" is in the content OR =obstacle_click_selector= is
      set → =_try_rememberme_reauth(page, profile_selector=...)=. The
      graduated selector is *prepended* to the candidate list so it runs
      first; falls through to hardcoded candidates
      (=[data-tracking-control-name*='rememberme']=, =.btn__primary--large=,
      =:has-text('Continue as')=, etc.).
   c. 3× 3s wait loop re-reading body.
   d. *Obstacle agent* (if wall persists AND =interaction_hints= present).
9. =_discover_job_selectors=.
10. Return result with optional =candidate_ready_selector=,
    =obstacle_click_winning=, =discovered_selectors=.

** The obstacle agent — =agents/obstacle_agent.py=
Per-call pydantic-ai =Agent= with =deps_type=_PageDeps= (wraps the live
Playwright Page). Created fresh per scrape — no shared state.

*** Model selection
- env =OBSTACLE_AGENT_MODEL= (first priority)
- fallback: =get_model("browser_scraper")= → =BROWSER_SCRAPER_MODEL= env →
  =CADDY_DEFAULT_MODEL= env → =openai:gpt-4o-mini=.

*** Tool surface — intentionally tiny
| Tool                   | Purpose                                   |
|------------------------+-------------------------------------------|
| =try_click(selector)=  | Click; rejects disallowed actions.        |
| =get_text()=           | Re-read =body.inner_text= to confirm.     |

NO =fill=, NO =submit=, NO navigation. Keeps blast radius minimal while
still solving click-the-right-button class obstacles.

*** Safety rails
- =DISALLOWED_ACTION_WORDS=: =sign out=, =logout=, =cancel=, =close=,
  =dismiss=, =back=. Checked on *both* the selector string AND the
  element's =inner_text= before clicking.
- =max_clicks=3= — stops runaway agents.
- Page text truncated to 4KB before being handed to the model.

*** Return contract
=run_obstacle_agent()= → ={resolved: bool, actions: [selectors], note: str}=.
=resolved= means *at least one click landed*; the *caller* must re-check
=_detect_login_wall(content)= to confirm the obstacle actually cleared.

** Probation gate — graduating agent knowledge into deterministic selectors
Shared helper in =scripts/hold_poller.py=: =_apply_probation_gate(...)=.

Flow: candidate must be observed in *N consecutive* successful scrapes
before it's promoted from =_X_candidate= to =X= in =css_selectors=. A
different candidate resets the counter to 1 (not zero — the new
observation already counts as match #1).

Two concrete applications:

| Candidate field                | Graduated field            | N | Source                                              |
|--------------------------------+----------------------------+---+-----------------------------------------------------|
| =_ready_selector_candidate=    | =ready_selector=           | 2 | first matching entry in =_JOB_SELECTOR_CANDIDATES["title"]= |
| =_obstacle_click_candidate=    | =obstacle_click_selector=  | 2 | last action in agent's =actions= list, iff =_detect_login_wall(content_after)= clears |

Bare-tag candidates (=h1=, =div=) are rejected upstream in
=scrape_page= — they match too promiscuously and would falsely harden
on login walls.

Threshold constants: =READY_SELECTOR_PROBATION_THRESHOLD=,
=OBSTACLE_CLICK_PROBATION_THRESHOLD= (both default 2).

Once graduated, repeat obstacles skip the LLM entirely —
=_try_rememberme_reauth= tries the graduated selector *first*.

** Cost model
| Scenario                                 | Per-scrape LLM calls                       |
|------------------------------------------+--------------------------------------------|
| Golden path (no obstacle)                | 0 (poller skips extractor LLM anyway)      |
| Known obstacle, graduated                | 0 (deterministic click)                    |
| Known obstacle, in probation             | 1 obstacle-agent call per scrape           |
| Novel obstacle on a hinted host          | 1 obstacle-agent call                      |
| Novel obstacle, no hints                 | 0 — returns =login_wall_detected= as before |

With Haiku at obstacle-agent model: ~$0.001 per invocation. Two probation
scrapes → ~$0.002 to harden a new obstacle permanently.

** Proxy support — SOCKS5 / HTTP via env
Both engines read proxy config from env in =lib/browser/engine.py=
=_get_proxy_config()=. Applied at browser-launch time (Camoufox:
=AsyncCamoufox(proxy=...)=; Chromium: =pw.chromium.launch(proxy=...)=).

| Env var                   | Example                         |
|---------------------------+---------------------------------|
| =BROWSER_PROXY_SERVER=    | =socks5://localhost:1080=       |
| =BROWSER_PROXY_USERNAME=  | (optional)                      |
| =BROWSER_PROXY_PASSWORD=  | (optional)                      |
| =BROWSER_PROXY_BYPASS=    | =*.internal,localhost=          |

*Caveat — Chromium does not honor auth on SOCKS proxies.*
Chromium-side limitation, not Playwright's. With =--engine chrome= +
authed SOCKS5, a warning logs and the credentials are ignored; the
server is still used but unauthenticated. Firefox / Camoufox handle
SOCKS5 auth natively, so the default engine Just Works.

Workarounds for Chrome-engine hosts (e.g. ARM/Pi via =--engine chrome=):
1. Use an HTTP proxy instead (Chromium supports HTTP-proxy auth fine).
2. Run a local proxy-chain, e.g. =gost -L=http://:8080
   -F=socks5://user:pass@localhost:1080=, then point
   =BROWSER_PROXY_SERVER= at =http://localhost:8080=.

** URL unwrapping — tracker/shortener layer
Upstream of everything else: =lib/url_unwrap.py= =unwrap_url(url)= runs in
the poller right after fetching =attrs.get("url")=. Pulls the real

=destination=, =q=, …), handles urlencoded and base64 targets,
recursive up to depth 5. Pure string parse — no network, so Pi-hole
never sees the tracker host.

Known unhandled: opaque shorteners (=bit.ly=, =lnkd.in=) that don't
embed the target in the query string. Would need a HEAD+follow-redirects
pass against a non-Pi-hole resolver; deferred.

* Scrape soft-fail + 301 link rewrite [2026-04-20]
:PROPERTIES:
:branches-future: api:feature/scrape-soft-fail-markers, api:feature/scrape-301-link-rewrite
:END:

** Problem
Scrape returns HTTP 200 but the page body says 'Expired Job' / 'this
posting has been filled' / etc. Scraper happily extracts nothing
useful, produces a stub JobPost, the hold-poller keeps retrying or
scoring runs against garbage.

Example: sendgrid email link redirects to hiring.cafe/viewjob/... →
page renders 'Expired Job' with 200 OK.

** Part A — soft-fail markers per hostname
Add jsonb field to scrape_profile:

  soft_fail_markers = models.JSONField(default=list, blank=True)
  # List of regex or literal strings. If any match the extracted body,
  # flip scrape.status='failed' with error_reason='expired'.

Django's JSONField on Postgres is jsonb by default (since 3.1), so GIN
index on the field is available if we ever need to search across
profiles for 'who has this marker'.

Analyzer agent populates markers semi-automatically:
- First encounter: human-in-the-loop via scrape_profile admin edit
  page. Seed generic list ('expired', 'no longer available',
  'position filled') but require manual confirmation to avoid
  blacklisting real descriptions that mention those words.
- Repeat hostname + short body + contains red-flag phrase → queue
  for human review, not auto-add.

Extractor check runs post-extraction; if marker hits, scrape fails
soft and the post isn't enriched.

** Part B — 301 link rewrite
The email contained a sendgrid click-tracker URL:
  https://u52508838.ct.sendgrid.net/ls/click?upn=... → 301 →
  https://hiring.cafe/viewjob/mm42v3y01518rygq

We currently store the sendgrid URL as job_post.link. When the user
clicks 'Run scrape' later it hits sendgrid again, which may 301 again
or expire outright.

Solution: on successful scrape, if the final URL (after redirect
chain) differs from the originally stored link, UPDATE
job_post.link to the canonical final URL. Scraper already knows the
final URL — just plumb it into the Scrape row's  field and, on
first successful extraction, write it back to the JobPost.

Note: avoid clobbering link if JobPost was manually edited. A
'link_canonical' attribute might make more sense than overwriting
the user-supplied link. TBD.

** Out of scope for now
- Upstream: tracking 'dead lead from email' signal back to the email

* Resume Word-export jinja audit [2026-04-20]                      :IMPORTANT:
:PROPERTIES:
:CREATED:  [2026-04-20]
:END:

Audit of the jinja markup captured in =todo.org= (Inbox, under "This is the
jinja markup for the word template") against the context dict that
=lib/services/resume_export_service.py= actually renders. Template file on
disk: =api/templates/resume_export.docx= (binary; the jinja in todo.org is the
transcribed copy of what's inside).

** Context source chain
=ResumeExportService.render_docx()= → =build_context()= → =Resume.to_export_context()=
→ post-processes =skills= into a dict grouped by skill_type → flattens the
grouped dict into top-level keys via =key_mapping= (Language → language_skills,
etc.).

Top-level keys in the rendered context:
- =header=, =HEADER= — both point to the same dict =({name, title, phone})=
- =summary=
- =experiences=  (list of dict)
- =educations=   (list of dict)
- =certifications= (list of dict)
- =projects=     (list of dict)
- =skills=       (dict {language_skills, database_skills, framework_skills, tool_skills, security_skills} — post-processed)
- =language_skills=, =database_skills=, =framework_skills=, =tool_skills=, =security_skills= — duplicated at top level so =skills.language_skills= and plain =language_skills= both work

** Gaps found

*** GAP 1: header.email / HEADER.EMAIL referenced but not in context
The jinja prints =({{HEADER.EMAIL}} * {{HEADER.PHONE}})= but
=Resume.to_export_context()= builds =header = {name, title, phone}= — email is
never added. Result: =HEADER.EMAIL= renders as empty, the " * " separator
shows with nothing to its left.

Fix: include =self.user.email= in the header dict. The User model already
carries the email.

*** GAP 2: Experience job title never rendered
=Experience.to_export_dict()= produces =title=, =start_date=, =end_date=,
=date_range=, =company=, =location=, =summary=, =descriptions=. The jinja
only uses =date_range=, =company=, =location=, =summary=, and =descriptions=
— the job title is silently dropped.

This is user-visible on the exported Word doc: every experience block shows a
company but no job title. A resume bullet like
  "2022 – Present — Acme — Austin, TX"
instead of
  "Senior Platform Engineer — Acme — Austin, TX (2022 – Present)".

Fix: add =({{ experience.title }})= (or equivalent) to the jinja experience
header line. No data-model change needed.

*** GAP 3: Project title + dates never rendered
=Project.to_export_dict()= produces =title=, =start_date=, =end_date=,
=is_active=, =description= (list). The jinja only iterates =proj.description=
for bullets — the project's name/title and dates are dropped. Every project
shows up as a naked bullet list.

Fix: add =({{ proj.title }})= (and optionally the date_range) to the Projects
section header.

*** GAP 4: Experience.summary uses "RP Summary" paragraph style
The jinja comment =({# This should use "RP Summary" style #})= flags that the
experience summary paragraph should be rendered with a specific Word
paragraph style ("RP Summary"), which docxtpl can't set from template
expression alone — needs a styled run/paragraph in the .docx template itself.
This is a template-authoring task, not a data-shape bug; noted here so the
owner knows the intent.

Same for projects: =({# This should use "RP Bulleted List" style #})= on the
bullet loop.

*** GAP 5: Skill-type string mismatch (dormant)
=Resume.tool_skills= property calls =self.skills_by_type("Tools/Platform")=
(plural "Tools"), but the canonical enum in =ingest_resume.SkillTag= is
=TOOL_PLATFORM = "Tool/Platform"= (singular), and the docx export's
=key_mapping= uses the singular form too. Nothing currently reads
=Resume.tool_skills= (no other callers in the repo), so this is dead code
rather than a visible bug. Either delete the unused =tool_skills= /
=language_skills= / etc. properties on =Resume=, or fix the string so the
property actually returns Tool/Platform skills.

*** Non-gap: certifications / educations are fully covered
The jinja fields =({title, issuer, issue_date, content})= and =({institution,
degree, major, minor, issue_date})= match =to_export_dict()= exactly. No
changes needed there.

** Recommended follow-ups (one TODO each, backend-only)

1. Add =email= to =Resume.to_export_context().header= so =HEADER.EMAIL=
   renders. Trivial. Probably a few lines in =resume.py=.
2. Update =api/templates/resume_export.docx= so the Experience block prints
   the job title and the Project block prints the project title + dates.
   This is a manual Word-template edit — touch the .docx in Word / LibreOffice,
   add ={{ experience.title }}= and ={{ proj.title }}= placeholders with the
   right paragraph styles, commit the updated binary.
3. Retire =Resume.tool_skills= / =language_skills= / =database_skills= /
   =framework_skills= / =security_skills= properties (no callers) OR fix the
   =Tools/Platform= → =Tool/Platform= string mismatch. Low priority — pick
   deletion unless a caller appears.

"""Map a Career Caddy Resume to Reactive Resume's JSON schema.

rxresume's import format is documented at
https://docs.rxresu.me/meta/structure — main sections we fill:
  basics { name, headline, email, phone, location }
  sections.{summary, skills, experience, education, certifications,
            projects}
"""
from job_hunting.models import Resume


def to_rxresume_json(resume: Resume) -> dict:
    ctx = resume.to_export_context()
    header = ctx["header"]
    basics = {
        "name": header.get("name") or "",
        "headline": header.get("title") or "",
        "email": getattr(resume.user, "email", "") or "",
        "phone": header.get("phone") or "",
        "location": "",  # TODO: plumb from Profile when available
    }
    summary = {"name": "Summary", "content": ctx.get("summary", "")}

    experience = {
        "name": "Experience",
        "items": [
            {
                "position": e.get("title") or "",
                "company": e.get("company") or "",
                "location": e.get("location") or "",
                "date": e.get("date_range") or "",
                "summary": e.get("summary") or "",
                "highlights": e.get("descriptions") or [],
            }
            for e in ctx["experiences"]
        ],
    }

    education = {
        "name": "Education",
        "items": [
            {
                "institution": e.get("institution") or "",
                "studyType": e.get("degree") or "",
                "area": e.get("major") or "",
                "date": e.get("issue_date") or "",
            }
            for e in ctx["educations"]
        ],
    }

    certifications = {
        "name": "Certifications",
        "items": [
            {
                "name": c.get("title") or "",
                "issuer": c.get("issuer") or "",
                "date": c.get("issue_date") or "",
                "summary": c.get("content") or "",
            }
            for c in ctx["certifications"]
        ],
    }

    projects = {
        "name": "Projects",
        "items": [
            {
                "name": p.get("title") or "",
                "date": "",
                "highlights": p.get("description") or [],
            }
            for p in ctx["projects"]
        ],
    }

    # Skills: rxresume expects items with {name, level, keywords[]}.
    # Our skills come out of build_context as a grouped dict.
    skills_items = []
    for skill in ctx.get("skills", []) or []:
        if isinstance(skill, dict):
            skills_items.append(
                {"name": skill.get("text") or "", "level": "", "keywords": []}
            )

    return {
        "basics": basics,
        "sections": {
            "summary": summary,
            "skills": {"name": "Skills", "items": skills_items},
            "experience": experience,
            "education": education,
            "certifications": certifications,
            "projects": projects,
        },
        "metadata": {
            "template": "azurill",  # default; per-resume override later
            "page": {"format": "letter", "margin": 18},
            "locale": "en-US",
        },
    }
#+end_src

** Step 4 — Client

#+begin_src python
import httpx
from django.conf import settings


class RxResumeClient:
    def __init__(self, base_url: str | None = None, api_key: str | None = None, timeout: float = 10.0):
        self.base_url = (base_url or settings.RXRESUME_BASE_URL).rstrip("/")
        self.api_key = api_key or settings.RXRESUME_API_KEY
        self._client = httpx.Client(
            base_url=self.base_url,
            headers={"x-api-key": self.api_key},
            timeout=timeout,
        )

    def import_resume(self, data: dict) -> str:
        r = self._client.post("/api/resume/import", json={"data": data})
        r.raise_for_status()
        return r.json()["id"]

    def print_pdf(self, resume_id: str) -> bytes:
        r = self._client.get(f"/api/resume/print/{resume_id}")
        r.raise_for_status()
        return r.content

    def delete(self, resume_id: str) -> None:
        # best-effort cleanup; ignore failures
        try:
            self._client.delete(f"/api/resume/{resume_id}")
        except Exception:
            pass

    def render(self, data: dict) -> bytes:
        rxid = self.import_resume(data)
        try:
            return self.print_pdf(rxid)
        finally:
            self.delete(rxid)
#+end_src

** Auto-score daemon (=ai/scripts/score_poller.py=)

Mirrors =hold_poller.py= shape. Loop:
1. =GET /api/v1/scrapes/?filter[status]=completed&filter[has_score]=false&per_page=N=
2. For each row, extract =job_post_id= from the relationship envelope
3. =POST /api/v1/scores/= with ={job_post_id}=
4. Respect per-run =--limit= (default 5) and =--interval= (default 30 min)
5. Per-user daily cost cap: before calling, =GET /api/v1/ai-spend/today/= —
   abort if over budget; log + emit a user-visible flash on next login

Gated by =CC_AUTOSCORE_ENABLED=1= on the service; per-user opt-in via a
=auto_score= bool on =Profile= (see API changes). Skipped posts are
counted, not retried silently — the user should see "12 posts skipped
this week: daily cap hit" in an admin/settings view.

** Shared URL extractor (=ai/lib/url_extractor.py=)

Port =career_caddy_automation/src/agents/url_extractor.py= verbatim — already
pure-Python (httpx + regex, no LLM dep). Canonicalizes tracker URLs, strips
=utm_*= / =trackingId= / =gclid= / etc. The existing manual paste + scrape
endpoints call it on whatever URL the user submitted. Closes the Inbox TODO
"Canonicalize tracker URLs on scrape ingest."

This module is deliberately /not/ shaped around email. URL in → canonical
URL out. Anything that feeds it URLs (paste page, chat scrape tool,
=career_caddy_automation/=) benefits equally.

1. =GET /api/v1/scrapes/?filter[has_score]=false= — join to =Score= via

2. =Profile.auto_score= boolean (default =false=, opt-in). One migration,

3. Per-user daily AI-spend endpoint =GET /api/v1/ai-spend/today/= — already

** What could go wrong

- /Cost blowout/ — the daily cost cap is the only safeguard. Must be
  server-enforced, not just honored by the daemon. Wire the cap into
  =ScoreViewSet.create= as a pre-check so any consumer (automation repo,

- /Silent skips/ — users deserve to see "we skipped 12 posts this week

* Subnav: inline render instead of in-element portal [2026-04-21]   :IMPORTANT:
:PROPERTIES:
:CREATED:  [2026-04-21]
:END:

Removed =#subnav-portal= indirection. =RouteLayout= now renders
=<Menus::SubnavBar>= directly; =main-application.hbs= lost its portal
=<div>= and CSS lost the =:not(:last-child) { display: none }= dedup hack
(audited — no nested =RouteLayout= users, so the hack was unused in
practice).

** Why
Portal rendering via ~{{#in-element}}~ runs =on-insert= modifiers *before*
the inserted node is attached to the live DOM in Firefox. Any
=ResizeObserver= attached that way read a zero-height =getBoundingClientRect=
and never corrected itself. Inline render mounts the bar in-place, so
modifiers fire against a live element in every browser.

** Other wins
- =@noSubnav= now means "no DOM," not "empty portal taking 0 height."
- Scroll-padding flow: =Menus::SubnavBar= writes =--subnav-h= on =<html>=
  via ResizeObserver; =app.css= sets =scroll-padding-top: var(--subnav-h, 4.5rem)=
  on both =html= AND =.course-main= (the actual scroll container —
  html/body are =100vh; overflow:hidden=, so scrolling happens inside
  =.course-main=, not on the document). Anchor jumps + =scrollIntoView=
  both clear the sticky bar exactly.

** Future guard
If anyone ever nests =<RouteLayout>= inside another =<RouteLayout>=, they
need =@noSubnav={{true}}= on the inner one or we'll render two bars
stacked. Comment left in =route-layout.hbs= noting this.
