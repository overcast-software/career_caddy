*** API notes [api]
No new FK column needed on any model. Every endpoint already has a user reference; the field name just varies.
Enforce at the queryset level per ViewSet using a class attribute =user_lookup= that names the ORM filter expression:

| ViewSet | Existing field | queryset filter |
|---|---|---|
| =CoverLetterViewSet= | =user= | =user_id=request.user.id= |
| =JobApplicationViewSet= | =user= | =user_id=request.user.id= |
| =ResumeViewSet= | =user= | =user_id=request.user.id= |
| =ScoreViewSet= | =user= | =user_id=request.user.id= |
| =SummaryViewSet= | =user= | =user_id=request.user.id= |
| =ProjectViewSet= | =user= | =user_id=request.user.id= |
| =JobPostViewSet= | =created_by= | =created_by_id=request.user.id= |
| =QuestionViewSet= | =created_by= | =created_by_id=request.user.id= |
| =AnswerViewSet= | none direct | =question__created_by_id=request.user.id= |
| =ScrapeViewSet= | none direct | =job_post__created_by_id=request.user.id= |
| =CompanyViewSet= | none | intentionally shared — no filter |
| Skill, Status | none | global lookup tables — no filter |
| Pivot tables (ResumeSKill etc.) | none | scoped via parent endpoint only |

Mixin approach (one place, covers most ViewSets):
#+BEGIN_SRC python
class OwnedQuerysetMixin:
    user_lookup = "user_id"          # override per ViewSet where needed
    def get_queryset(self):
        qs = super().get_queryset()
        if self.request.user.is_staff:
            return qs
        return qs.filter(**{self.user_lookup: self.request.user.id})
#+END_SRC

ViewSets with a different field name just set =user_lookup = "created_by_id"= or ="question__created_by_id"=.
No model migrations. No new columns.

** Pydantic-AI toolsets — dynamic tool scoping [2026-04-10]
[[file:todo.org::*MCP exposure][↑ todo.org]]

*** Problem
All pydantic-ai agents connect to =career_caddy_server.py= via =MCPServerStdio=, spawning a full Python subprocess per agent run. Every agent gets the same 14 tools regardless of what it actually needs. This adds latency, memory overhead, and prevents fine-grained tool scoping.

*** Solution: =CareerCaddyToolset(AbstractToolset)=
A single custom toolset class with a =scope= parameter that selects which =lib/api_tools.py= functions to expose. No subprocess, no MCP protocol overhead — direct in-process calls.

#+BEGIN_SRC python
from lib.toolsets import CareerCaddyToolset, CareerCaddyDeps

# Scoped to just what this agent needs
agent = Agent(
    "openai:gpt-4o-mini",
    deps_type=CareerCaddyDeps,
    toolsets=[CareerCaddyToolset(scope="scoring")],
)
result = await agent.run(prompt, deps=CareerCaddyDeps(api_token=token))
#+END_SRC

*** Deps type
#+BEGIN_SRC python
@dataclass
class CareerCaddyDeps:
    api_token: str
    base_url: str = "http://localhost:8000"
#+END_SRC

The toolset builds an =ApiClient(deps.base_url, deps.api_token)= from =ctx.deps= at call time and passes it to each =api_tools.*= function.

*** Scopes
| Scope | Tools | Use case |
|---|---|---|
| =all= | All 19 functions | Full access |
| =career_caddy= | Original 14 | Drop-in replacement for current MCPServerStdio |
| =job_discovery= | company search/create, job post create/find/search | Email pipeline agent |
| =scoring= | get_job_posts, get_career_data, score_job_post, get_scores | Scoring agent |
| =application_tracking= | job application CRUD, update_job_post, find_by_link | Status update agent |
| =scrape_management= | create_scrape, get_scrapes, update_scrape | Scrape queue agent |

Custom tool lists also accepted: =CareerCaddyToolset(scope=["score_job_post", "get_career_data"])=

*** Implementation: =ai/lib/toolsets.py= (new file)
Contains:
- =CareerCaddyDeps= dataclass
- =TOOL_REGISTRY= — dict mapping tool names to =api_tools.*= functions
- =SCOPES= — dict mapping scope names to sets of tool names
- =_make_tool_wrapper(fn)= — builds a =RunContext=-aware wrapper that preserves the original function's signature (minus the =api= param) for pydantic-ai schema generation
- =CareerCaddyToolset(AbstractToolset[CareerCaddyDeps])= — delegates to internal =FunctionToolset=
- Factory functions: =job_discovery_toolset()=, =scoring_toolset()=, etc.

*** Key challenge: signature preservation
Pydantic-ai uses =inspect.signature()= to build JSON schemas for tool parameters. The wrapper must copy the original function's parameter list (minus =api: ApiClient=) and construct a new =__signature__=. Same pattern FastMCP uses internally for =@server.tool()=.

*** Migration path
1. Create =ai/lib/toolsets.py=
2. Update =career_caddy_agent.py=: replace =MCPServerStdio= with =CareerCaddyToolset(scope="career_caddy")=, add =deps_type=CareerCaddyDeps=
3. Update =job_email_to_caddy.py=: indirect — it calls =add_job_post()= which uses the agent
4. =email_classifier_agent.py=: no change (uses email server only)
5. =job_extractor_agent.py=: no change (pure LLM, no tools)

*** Security: browser credentials are unreachable
- =lib/api_tools.py= imports only =httpx=, =json=, =pydantic= — zero knowledge of =secrets.yml= or browser automation
- =lib/toolsets.py= imports only from =lib/api_tools.py= and =pydantic_ai= — MUST NOT import from =lib/browser/*= or =mcp_servers/browser_server.py=
- =CareerCaddyDeps= holds only =api_token= + =base_url= — no credential paths
- No tool in the registry can read =secrets.yml=, trigger a browser login, or access notmuch email
- Browser/email capabilities live exclusively in =browser_server.py= and =email_server.py=, which are never imported by the toolset
- This is the same isolation model as =public_server.py= — enforced by import boundaries, not runtime checks

*** What stays the same
- =career_caddy_server.py= / =public_server.py= / =gateway.py= — still serve external MCP clients
- =lib/api_tools.py= — the single source of truth for tool implementations
- Email and browser MCP servers — separate tool sources, outside this scope

*** Agent identity awareness [2026-04-11]
Two paths for the agent to know the user's identity:

**** Web chat (chat_server.py) — JWT pass-through (Option C)
- Frontend sends JWT in =Authorization: Bearer= header to =/api/v1/chat/=
- Django proxy validates the JWT, then forwards the *same token* to the chat service
- No temporary API keys created — eliminates the DB write per message overhead
- Chat service calls =GET /api/v1/me/= with the forwarded token to fetch user profile
- Profile (name, email, linkedin, github) is injected into the system prompt
- Same token used for all downstream tool calls via =ApiClient=
- System prompt explicitly instructs the agent to use this data and never deny access
- Name fallback chain: =first_name last_name= → =username= → =email= → "authenticated but no profile"
- Chat service accepts both JWTs and =jh_*= API keys — the API's auth stack handles either

**** Public MCP server (public_server.py)
- FastMCP =instructions= parameter tells MCP clients to call =get_current_user= at session start
- =get_current_user= tool returns the authenticated user's full profile from the token claims
- No system prompt injection needed — the MCP client's own LLM handles the instructions
- Auth: =jh_*= API keys provisioned by the user (long-lived, for CLI/Slack/third-party apps)

**** Key insight
The web chat injects profile into the system prompt (server-side). The MCP server provides
instructions for the client to fetch it (client-side). Both paths achieve the same result:
the agent knows who the user is from the first message.

Two auth token types serve different clients:
- JWT: browser sessions (short-lived, auto-refreshed by the SPA)
- API key (=jh_*=): MCP clients, Slack, CLI, third-party apps (long-lived, user-provisioned)

*Optimization strategies (future):*
1. *On-demand browser-mcp* — don't run it 24/7. Use =docker compose up browser-mcp= only when scraping, or set =restart: "no"= and have the API start it via Docker API before dispatching. Scraping is infrequent; the browser idles 99% of the time.
2. *Merge chat + browser-mcp into one process* — both are Python/FastMCP on the AI image. A single service could host both servers on different ports (or one port with path routing). Saves ~256MB+ RAM from the duplicate Python runtime and Camoufox memory.
3. *Slim AI image variants* — build two images: =ai-lite= (no camoufox, for mcp + chat) and =ai-browser= (with camoufox, for browser-mcp only). The =INSTALL_CAMOUFOX= build arg already supports this.
4. *Move browser-mcp off the VPS* — run it locally or on a beefier machine. The hold-poller pattern already supports this (queue on VPS, scrape locally). Only direct =/scrapes/new= would need the VPS browser.
5. *Scale down replicas* — reduce =GUNICORN_WORKERS= from 2 to 1 if traffic is low. The chat and MCP services are single-worker already.
6. *Swap Camoufox for lightweight scraping* — for sites that don't need JS rendering or anti-bot evasion, use =httpx= + BeautifulSoup directly in the API (Tier 0 in the scrape profile system). Reserve Camoufox for login-wall sites only.

*** Running lint
#+begin_src bash

npm --prefix /path/to/frontend run lint:js     # just ESLint
npm --prefix /path/to/frontend run lint:hbs    # just template lint

#+end_src

*** Fixing prettier issues
Prettier must be run from within the frontend submodule context so that =prettier-plugin-ember-template-tag= resolves correctly. Running prettier from the repo root will fail because the root =noop.js= can't find the plugin.

#+begin_src bash
# Check what's wrong:
npm --prefix /path/to/frontend run lint:format

# Auto-fix (pass files after -- --write):
npm --prefix /path/to/frontend run lint:format -- --write /absolute/path/to/file.hbs
#+end_src

*** Key gotcha
Do NOT run =npx prettier= from the repo root — it picks up the root config and fails to resolve the Ember template tag plugin. Always use =npm --prefix frontend run lint:format=.

Single linter: ruff.

* Color Palette System [2026-04-11]

** Architecture
Two independent dimensions: *mode* (light/dark) and *palette* (indigo/blue/jade/rose/amber/violet).
Applied via =data-theme= and =data-palette= attributes on =<html>=.
ThemeService stores both in localStorage (=theme-mode=, =theme-palette=).
Settings > Appearance tab lets users pick palette + toggle dark mode.

** How to add a new palette

1. Pick a color family (Tailwind reference or custom). You need shades 50-900.
2. Convert hex to space-separated RGB: =printf "%d %d %d\n" 0xRR 0xGG 0xBB=
3. Add a CSS block in =frontend/app/styles/app.css= after existing palette blocks:
   #+begin_src css
   [data-palette="teal"] {
     --accent-50:  240 253 250;
     --accent-100: 204 251 241;
     /* ... shades 200-800 ... */
     --accent-900: 19 78 74;
     --accent: rgb(var(--accent-500));
     --tee-bg: rgb(var(--accent-200));
     --page-bg: #f0fdfa;
   }
   #+end_src
4. Add the name to =PALETTES= array in =frontend/app/services/theme.js=.
5. Add swatch colors to =SWATCH_COLORS= in =frontend/app/controllers/settings/appearance.js=.
6. No template changes needed — all templates use =accent-*= classes.
7. Test in both light and dark modes.

*** How omarchy themes work

* AI Chat — Architecture Reference [2026-04-13]
New agents: read this section to understand how AI chat works end-to-end.

** Overview
Chat is a three-layer streaming system: Ember frontend → Django API proxy → Pydantic-AI chat server.
The user types a message, it flows through all three layers, and the response streams back as SSE events.
The AI agent has access to 27 tools that CRUD the user's career data via the same API.

** Message flow (end-to-end)
#+begin_src
User types in frontend (Chat::Panel textarea)
  ↓
ChatService.sendMessage(text)                    [frontend/app/services/chat.js]
  - adds user message + empty assistant placeholder to @tracked messages array
  - calls session.ensureFreshToken(90) to ensure valid JWT
  - POST ${baseUrl}chat/ with JSON body:
    {message, history, conversation_id, page_context}
    Authorization: Bearer <jwt>
  ↓
chat_proxy(request)                              [api/job_hunting/api/chat.py]
  - validates JWT via DRF JWTAuthentication
  - extracts raw token string from Authorization header
  - builds payload: {message, token, history, conversation_id, page_context}
  - streams POST to http://localhost:8031/chat via httpx.Client.stream()
  - returns StreamingHttpResponse (text/event-stream)
  ↓
chat(request)                                    [ai/mcp_servers/chat_server.py]
  - _fetch_user_profile(token): GET /api/v1/me/ → name, email, linkedin, github
  - _build_agent(user_profile, page_context): creates pydantic-ai Agent
    with dynamic system prompt (profile + page context injected)
  - CareerCaddyDeps(api_token=token, base_url) passed to agent.run_stream()
  - agent streams text chunks + makes tool calls as needed
  - each tool uses the same JWT for API calls → data is scoped to the user
  ↓
SSE event stream yields back through all layers:
  data: {"type": "text", "content": "chunk..."}       ← partial text
  data: {"type": "navigate", "url": "/resumes"}       ← programmatic navigation
  data: {"type": "done", "content": "full text",      ← final response
         "conversation_id": "uuid", "usage": {...}}
  data: {"type": "error", "content": "error msg"}     ← on failure
  ↓
Frontend SSE reader accumulates text chunks, updates last message,
handles navigate events, stores conversation_id for multi-turn.
#+end_src

** History management
- Frontend sends last 20 messages (excluding current turn) as =history= array
- Chat server converts to pydantic-ai =ModelRequest=/=ModelResponse= objects
- =truncate_message_history=: drops old messages when history exceeds ~20k tokens (estimated at 3 chars/token)
- =sanitize_orphaned_tool_calls=: removes tool call/return pairs where not all parallel calls were answered (handles MCP crashes and partial failures)
- Tool response cap: 300k chars max per individual tool response (prevents context blowout)

** Frontend chat features
*** Sidebar vs full-page
- Sidebar: =<Chat::Sidebar>= wraps =<Chat::Panel>=, appears as CSS grid area =course-chat= (24rem wide)
- Full-page: =/caddy= route renders =<Chat::Panel>= in =<RouteLayout>= main area
- Both share the singleton =chat= service — messages persist across route transitions
- Sidebar and nav sidebar are mutually exclusive (opening one closes the other)

*** Markdown rendering
- Assistant messages render through =<MarkdownToHtml>= (ember-cli-showdown)
- User messages render as plain =whitespace-pre-wrap= text
- CSS class =.chat-markdown= provides compact styling (tighter than =.markdown-content=)

*** SPA link navigation
=chat/message.js= intercepts clicks on =<a href>= inside assistant messages.
Relative paths route through =router.transitionTo()= — no page reload.
External links (=http=, =mailto:=) pass through normally.

*** Programmatic navigation
Two mechanisms:
1. AI includes =<!-- navigate:/path -->= in text. Frontend regex strips it and calls =router.transitionTo()=.
2. Server emits ={type: "navigate", url: "/path"}= SSE event. Frontend handles it in the SSE reader.
Both navigate only on clear user intent ("take me to", "show me").

*** Elicitation buttons (Phase 1)
AI emits a =```json= code block at end of response: ={elicitation: true, actions: [{label, message}]}=.
=chat/message.js= detects via regex, renders accent-colored buttons below markdown.
Clicking sends =action.message= as a new user chat message. Max 3 actions.

*** Page context
=main-application.js= sets =chat.currentPage = {route, url, params}= on every route change.
Sent in the chat POST body. Chat server injects "Current Page" section into system prompt.
Enables the AI to understand "this job post" or check data on the current page.

*** Auth boundary messages
- 401 (not logged in): friendly markdown message with [login](/login) link
- 403 (guest account): markdown message suggesting signup with [login](/login) link
- Both use SPA link navigation (no page reload)

*** Streaming UX
- "Thinking..." pulse animation while =isStreaming= is true
- Text accumulates chunk by chunk, message updates in place
- Auto-scroll to bottom after each chunk

** Usage tracking
- =report_usage()= fires async after each chat response (fire-and-forget)
- POST to =/api/v1/ai-usages/= with: agent_name="career_caddy_chat", model_name, trigger="chat", token counts
- =AiUsage= model: user FK, agent_name, model_name, trigger, request/response/total tokens, estimated_cost_usd
- =AiUsageViewSet=: non-staff sees own records only, staff can filter by =?user_id==

- =chat_server.py= MUST NOT import =email_server=, =browser_server=, or =lib/browser/*=
- No =CC_API_TOKEN= env var on the chat service — auth comes from the calling proxy
- No =secrets.yml= or mail directory access

*** Option A — DB-stored configs with cached API fetch ⟵ RECOMMENDED

* Hold Poller, Screenshots, Multi-Model Pipeline [2026-04-14]

** Motivation
The browser scraper agent is a pass-through that wastes LLM tokens — the LLM just calls =scrape_page(url)= and echoes the result. By calling =scrape_page()= directly as a Python function and only using LLMs for the intelligent steps (job extraction via Sonnet, CRUD via gpt-4o-mini), we save ~56% per scrape.

Browser scraping on credentialed sites fails silently in prod Docker (no sessions/cookies). The hold-scrape pattern lets admin queue URLs from the prod UI, then a local poller processes them with a headed browser and saved sessions.

** Architecture

*** Multi-model pipeline (cost-optimized)
| Step                  | Model              | Cost     |
|-----------------------+--------------------+----------|
| =scrape_page()=       | None (direct call) | $0.00    |
| =extract_job_from_content()= | Claude Sonnet      | ~$0.10   |
| =add_job_post()=      | gpt-4o-mini        | ~$0.002  |
| *Total per scrape*    |                    | *~$0.10* |

*** Hold poller — scrape-only worker

*What it is:* A local process (not Docker) that polls the Career Caddy API for
scrapes with =status=hold=, runs Camoufox to fetch the page, and PATCHes the
content back. The API handles everything downstream (extraction, job post
creation, scrape profile updates).

*Why it exists:* Browser scraping is too heavy for the VPS. The poller runs on
a desktop or Pi, keeping the VPS as a thin API + frontend layer.

*Running it:*
| Method | Command |
|--------+-----------------------------|
| Standalone (prod) | =cd ai && cp .env.prod.example .env && uv run caddy-poller= |
| Standalone (local dev) | =cd ai && CC_API_BASE_URL=http://localhost:8000 CC_API_TOKEN=xxx uv run caddy-poller= |
| With dev stack | =make up-full= (Docker stack + poller in background) |

*Prerequisites:*
- =uv sync= + =uv run python -m camoufox fetch= (one-time)
- =CC_API_BASE_URL= and =CC_API_TOKEN= env vars
- No LLM keys needed on the worker — all LLM work happens API-side

*Backoff:* The poller uses exponential backoff: 2, 4, 8, 16, 32, 64, 128, 256
seconds. Resets to 2s when work is found. Stops after the full sequence with no
work (~510s total). This means you run it, it processes any hold scrapes, backs
off, and exits. Run it again when you queue more holds.

*Flow:*
#+BEGIN_SRC
caddy-poller starts
  poll: GET /api/v1/scrapes/?filter[status]=hold
  found work?
    yes → for each hold scrape:
            PATCH status → running
            scrape_page(url) — Camoufox, no LLM
              login wall? → status=failed + screenshot
              success? → PATCH job_content + status=completed
                → API fires _maybe_trigger_extraction():
                  parse_scrape() → LLM extraction (gpt-4o-mini)
                  _update_scrape_profile() → domain stats
                  _generate_profile_hints() → LLM hints (first time only)
            reset backoff to 2s
    no  → backoff: 2 → 4 → 8 → 16 → 32 → 64 → 128 → 256 → stop
#+END_SRC

| =BROWSER_HEADLESS= | =true= | Set =false= for VNC debugging (pibu has VNC) |

* Domain Scrape Profiles — learning loop for scrape cost reduction [2026-04-14]

** Motivation
The job extractor LLM call (~$0.025/scrape with Haiku) is the main per-scrape cost. For domains scraped frequently (LinkedIn, Indeed, Glassdoor), the LLM re-discovers the same page structure every time. A domain-level profile that captures what worked before could eliminate or reduce the LLM cost on repeat scrapes.

** How the learning loop works
1. First scrape of =linkedin.com= → no profile → full LLM extraction ($0.025)
2. Extraction succeeds → auto-populate profile from the result
   - LLM writes =extraction_hints=: "title in =.top-card-layout__title=, company in =.topcard__org-name-link=, salary format USD annual"
   - =avg_content_length= set from this scrape
   - =requires_auth= set from ScrapeStatus history
3. Next =linkedin.com= scrape → profile hints injected into extractor prompt → cheaper/faster
4. If extraction fails or quality degrades → fall back to full LLM, flag profile as stale
5. Staff can review/edit profiles in admin UI

** Tiered scrape cost structure
| Tier | When                          | Model          | Cost/scrape |
|------+-------------------------------+----------------+-------------|
|    0 | Known domain + CSS selectors  | None (determ.) | $0.00       |
|    1 | Known domain + hints          | gpt-4o-mini    | ~$0.005     |
|    2 | Unknown domain                | Haiku          | ~$0.025     |
|    3 | Complex / high-value page     | Sonnet         | ~$0.10      |

Tier selection can be automatic (profile exists? use it) or manual (admin chooses tier per scrape).

After a successful extraction, ask the LLM one follow-up question:

#+end_src
This is a one-time ~$0.01 cost that pays off over many future scrapes.

* Builder-form style conventions [2026-04-14]
Reference for agents working with =builder-form= in =frontend/app/styles/app.css=.

** Layout classes
| Class | Behaviour | Use for |
|-------+-----------+---------|
| =builder-form= | 2-col grid: =max-content 1fr= | Default form layout (label + input per row) |
| =builder-form.two-col= | Equal 2-col grid, fields side-by-side | Settings profile, questions form |
| =.field= | Direct child: =display:contents= (dissolves into grid) | Standard label+input pair |
| =.field-full= | =flex-direction:column=, spans both cols. *All* child =input/textarea= get =width:100%= | Full-width fields (address, description, question content) |
| =.field-row= | =display:flex=, spans both cols. Does *not* force child input width | Inline groups where you control widths with Tailwind |
| =.form-actions= | Button row at bottom, spans both cols | Save / Cancel / Delete |

** Key gotcha: =.field-full= forces =width:100%= on all descendant inputs
If you put a grid or flex row of multiple inputs inside =.field-full=, each input expands to 100% and stacks vertically. Use =.field-row= instead when you need multiple inputs on one line (e.g. the links editor on settings/profile).

** Inline styles
Never use inline =style="..."= attributes. Tailwind utilities are the standard. If a Tailwind class is being overridden by =builder-form= CSS, choose the correct form layout class (=field-row= vs =field-full=) rather than fighting it with inline styles.

** Read-only views

** Key pattern: JSON:API relationship =data= linkage
When a serializer declares to-many =relationships=, =BaseSerializer.to_resource= only emits
=links= (no =data=). This is correct for relationships that are paginated or expensive.
For relationships that should resolve from =included= sideloads, the serializer's
=to_resource= override must add =data= arrays:

#+BEGIN_SRC python
# In the serializer's to_resource():
for rel_name, rel_type in rel_defs:
    _, items = self.get_related(obj, rel_name)
    linkage = [{"type": rel_type, "id": str(item.id)} for item in items]
    res.setdefault("relationships", {})[rel_name] = {
        "data": linkage,
        "links": existing_links,
    }
#+END_SRC

Without =data=, Ember Data has no way to match =included= records to their parent —
even if the records are in the payload.

** Key pattern: async hasMany proxy resolution
Even with proper sideloading, Ember Data's =async: true= hasMany returns a =PromiseManyArray=.
Accessing =.length= before resolution returns 0. Routes must =await= relationships:

#+BEGIN_SRC javascript
const resume = await this.store.findRecord('resume', resume_id);
await Promise.all([resume.skills, resume.summaries, resume.projects]);
// Now .length and .objectAt() work synchronously in templates
#+END_SRC

This await resolves instantly from the store (no extra fetch) when data is sideloaded.
