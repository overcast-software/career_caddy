** Dockerfile caching
[[file:todo.org::*Caching deploy steps so it goes faster][↑ todo.org]]

- Standard optimisation: copy dependency manifests (=pyproject.toml=, =uv.lock= for API; =package.json= for frontend) before source code so the dependency layer is cached between rebuilds.
- Self-contained, zero risk, no design decisions required.
- Expected improvement: rebuilds after source-only changes drop from ~3 min to ~30 sec.

*** API notes [api]
- Current API Dockerfile copies the entire source tree before =uv sync=, busting the dependency cache on every code change.
- Fix: copy =pyproject.toml= + =uv.lock= -> run =uv sync --frozen --no-install-project= -> copy source -> run =uv sync= again. Two-stage install, cache-friendly.
- The =--no-install-project= flag installs dependencies only (not the app itself), so the heavy layer is cached as long as lockfile doesn't change.
- Dagger pipeline in =dagger/src/main.py= may also benefit from the same reordering.
- No risk to production behaviour; purely build-time change.

** Favorite answers not appearing in career-data [2026-04-10]
[[file:todo.org::*Favorite answers bug][↑ todo.org]]

*** Root cause
=CareerData.favorite_answers= (=api/job_hunting/lib/models/career_data.py:26-35=) first fetches questions where =question.favorite=True=, then fetches answers for those questions where =answer.favorite=True=. This means an answer only appears in career-data if *both* the question and the answer are favorited.

The frontend UI only exposes favoriting on answers (the star button on the answers index page says "Favorited answers appear in Career Data"). Users never favorite the parent question, so the query always returns an empty list.

*** Fix (implemented [2026-04-10])
Multi-layer fix across API and frontend:

**** Backend
- =CareerData.favorite_answers=: query answers directly by =answer.favorite=True= for the user
- =CareerData.question_applications=: derive from answers with favorited answers, not =question.favorite=
- =CareerData= endpoint: returns object refs (=resume_ids=, =cover_letter_ids=, =answer_ids=) in =meta=
- =AnswerService=: all 3 query sites now derive "favorite questions" from favorited answers
- =QuestionViewSet=: removed =favorite= from create/update whitelists (read-only)
- =AnswerViewSet=: added update/partial_update; syncs =question.favorite= as denormalized cache after save/delete

**** Frontend
- =career-data= model: =@tracked isDirty= flag, =markDirty()= method. Default is clean (from API). Dirty is write-only.
- =career-data/index= route: returns cached record if clean; fetches fresh if dirty or absent. =unloadAll= before re-fetch so the new record starts clean.
- All 8 favorite toggle sites call =this.store.peekRecord('career-data', '1')?.markDirty()= after save.

**** Design principle
Nobody directly favorites a question. A question is "favorite" only because it has a favorited answer. The =question.favorite= DB column is kept as a denormalized cache synced by the API on answer save/delete.
